# 中文内容叙事方法论（narrative-methodology · 通用）

> 通用、account-agnostic 的中文长文写作方法论，写作类 skill 在正文里引用本文件对应节。
> 与 `writing-constraints.md` 分工：constraints 是**质量下限/可机检规则**（不能做什么）；本文件是**叙事上层方法论**（怎么写得打动人）。
> 账号特定的声音、定位、对标、平台排版**不在这里**，由项目 `account-style.md`（见 `style-contract.schema.md`）与项目平台层承载。
> 本文示例用通用公开案例，不含任何账号私有信息。

## 0. 体裁判断（最前置）

本方法论适用于**故事 / 新闻 / 评论 / 趋势分析**类长文，**技术教程类不适用**。

一句话判断：读者读完是想「**被打动**」还是想「**学会做**」？想被打动 → 用本方法论；想学会做 → 走技术文档规范（重信息密度、trade-off、可复现步骤、cheatsheet）。

| 维度 | 故事 / 新闻 / 评论（本方法论） | 技术 / 教程 / 工具（另一套） |
| --- | --- | --- |
| 目标 | 让读者被打动、想转发 | 让读者学会、能复现 |
| 节奏 | 跌宕，每 3 分钟一高潮 | 平稳，重信息密度 |
| 重点 | 情绪、痛点、共鸣 | 正确性、可验证、可复现 |
| 开篇 | 第一人称代入、戏剧场景 | 直接说事、痛点+解法 |
| 中段 | 人物 + 引语 + 金句 + 数据 | 原理 + 步骤 + 代码 + 取舍 |
| 收尾 | 开放洞察 / 留白 / 反预期 | 总结 / cheatsheet / 推荐阅读 |
| 中立平衡 | 慎用（会软掉立场） | 必备（边界条件 / 版本约束） |
| 标题 | 钩子型、反预期、口语化 | 实测 / 教程 / 原理深度解析 |

混合体裁按主体裁的规则写：技术+故事钩子 → 仍按技术写，每段加场景钩子引入；新闻+技术深度 → 仍按新闻写，技术细节作为「为什么」的解释段。

## 1. 最高原则 · 命中读者痛点（先于一切）

一切技巧、结构、风格的目的都服务一件事——让读者在 3 秒内觉得「**这就是我**」。读者用快思考（系统 1）靠情绪决定要不要读、点赞、转发；理性的「哦原来如此」不带来转发，**只有情绪能**。

**动笔前必答 2 问**：
1. **痛点是什么？** 读者具体处于什么困境？
2. **主打哪类情绪？**（情绪四象限）
   - **焦虑制造**：让他更紧张（"95% 的人都失败了"）
   - **焦虑缓解**：让他松一口气（"不是你的错，是没人说清"）
   - **认同 / 愤怒**：让他想替自己出口气
   - **优越感 / 启蒙**：让他想转发给别人看（"我懂了别人不懂的事"）

只能让读者「理性认同」而无任何情绪反应的文章，不会被转发。

**默认结构**：制造痛点共鸣 → 用案例放大焦虑 → 给新视角解释为什么读者其实不必焦虑（或问题不在他）→ 给「我懂了」的释放感让他想转发。

**发布前 3 问**：① 3 秒入戏：第一段 3 秒内有没有「这就是我」？② 情绪类型：开头主打哪种情绪，说不清就重写。③ 转发动机：读完会想转发给谁、为什么，说不清就是没击中痛点。

## 2. 长文密度规则（与字数无关）

**长文不是问题，平淡才是问题。** 每约 600–800 字（3 分钟阅读 / 一屏滑动）必须有一个**惊喜点**。写完一段自查：「读到这一段的读者会不会坐直一点？」不会就改。

**五类惊喜点**：
- **反直觉断言**：把因果归到反常识的地方
- **故事转折**：先扬后抑 / 公开认错 / 一年后反转
- **扎心数据**：对比悬殊的真实数字（73% vs 23%、95% 零 ROI）
- **金句反转**：「不是 X 不行，是被它接进去的 Y 本来就不行」
- **画面感场景**：凌晨群发的裁员邮件、具体到人的细节

## 3. 段落结构 · 黄金模板

```
[痛点] 你也遇到过这种情况吧？……的时候……
[场景] 就拿我来说，上周……
[转折] 但你猜怎么着？/ 直到我发现……
[干货] 具体是这样的：它做了 A、B、C……（带数字）
[金句] 说白了，X 不是 Y，而是 Z。
```

段落饱满度：每个论点至少一整段（80–150 字）；只有一两句的段落要么合并要么展开说透；引用、数据、比喻至少占三分之一。

## 4. 文章结构 · 三幕剧 + 故事思维

故事/评论类长文按三幕剧行文（人脑对「故事」的接受度远高于「说明」；没有挫折的成功故事 = 炫耀 = 读者划走）：

```
第一幕 建置：熟悉的场景/痛点（共鸣）→ 大家通常怎么做（看似合理）→ 埋一个隐患
第二幕 对抗：痛点爆发、常规方式失效（绝境）→ 走弯路/踩坑/失败（跌宕）→ 找到真解法（转折）→ 解法的过程细节（干货）
第三幕 解决：前后对比（数据/体验）→ 从个人经验升华到普遍规律 → 最后一句值得截图转发
```

- **开头（前 3 段定生死）**：制造共鸣 + 建立期待 + 别废话（不自我介绍、不铺垫背景，直接切入）。
- **中间（故事+干货交替，必须有低谷）**：每个 Case 有「失败/踩坑」环节；用「场景→尝试→失败→转折→成功」而非「场景→成功」；穿插真实引用/截图；图表支撑叙事而非替代。
- **结尾（留一句值得转发的话）**：不复读全文；给更大的思考、从个人升华到普遍规律；最后一句值得截图。

## 5. 语气方法论（通用部分）

要做到：**口语化**（"你想想""说白了""但问题来了"）、**有情绪**、**有画面感**（不说"效率低"，说"大半天才勉强出一篇"）、**有观点**（不中立叙述，要判断）、**用具体数字**（数字比形容词有说服力）。

绝对不做：说明书式列功能、空洞修饰词（非常/很好地/高效地）、一句话一段（每段至少 3–5 句把一个点说透）、客套（"让我来介绍一下"）、被动语态（"该系统被设计为"→"它就是干这个的"）。

### AI 味的三大根源（写每段前自查，通用诊断）

AI 写中文的核心毛病是**怕**：
- **怕读者跟不上 → 多解释**：病灶是「先建对照基线 → 反转 → 解释」（X 看起来 Y 但其实 Z 因为 W）；改法是直接断言 W。
- **怕立场硬 → 用被动 / hedge**：「似乎」「好像」「被……绕开了」；改法换主动语态、敢断言。
- **怕用错 → 避成语 / 用日常词稀释**：金句位置用平淡日常词；改法是金句位置敢用成语和判断词（视而不见 / 阳奉阴违 / 不动声色）。

自查 4 问：这个铺垫删掉读者还跟得上吗（跟得上就别写）？这句能被动改主动吗？金句位置有没有密度更高的成语/判断词？这段读出声像聊天还是像新闻稿（像新闻稿就重写）？

### 比喻癖（单独强调）

模型有系统性比喻癖——在不该用的地方硬塞、或用属性不匹配的比喻撑主线。**写每个比喻前自问：喻体核心属性（可还/可见/可分/可拆…）跟被喻物匹配吗？** 不匹配就改直白陈述。**金句位置尤其慎用比喻**（出错率最高），首选直白判断 + 成语。

## 6. 标题公式

好标题：短（≤15 字最佳）、有悬念、有具体词。

常用模式：
- **反常识**：「程序员最大的谎言：我写个脚本就行」
- **数字 + 场景**：「我让 AI 替我打工 30 天」「每天 3 分钟掌握全局」
- **疑问句**：「为什么你的自动化永远做不成？」
- **实录 / 实测**：「实测：X」「X 实录」

禁忌：太长（超 20 字就删减）、太泛（"AI 时代的工作方式"→没人点）、太技术（"基于 Agent 框架的自动化工作流实践"→没人点）。

## 7. 图表使用

图表用来展示**对比、流程、结构**——文字说不清的东西；不用图表展示文字已说清的内容（纯装饰）。图表标题要有信息量（"图 1：从单机到云端的架构演进"✅ 而非"图 1：系统架构"❌）。具体图表形态（SVG / PNG / 是否适配某平台）由项目平台层决定。
