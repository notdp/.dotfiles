# Debug

## 1. 收集信息

如果 `$ARGUMENTS` 非空，将其作为 bug 描述的起点。否则交互式收集：

- 精确的错误信息 / 异常堆栈
- 复现步骤（或最小复现条件）
- 预期行为 vs 实际行为
- 最近的相关变更（`git log`）

修复开始前先找到复现路径，并把它固化成可重复运行的反馈环；无法复现时，当前产出是复现需求和证据缺口。

搜索时中英文关键词都尝试。不只搜索错误信息本身，还要搜索问题模式和类似案例。
仅当需要查询公开文档、外部 issue 或其他 Web 资源时才使用 WebSearch；本地代码、配置、环境问题优先使用 Grep / Read / 日志 / 复现定位。

## 2. 构造反馈环（硬门控）

调试的第一产物不是解释，而是 agent 可反复运行的 pass/fail signal。反馈环建立后再进入修复。

按成本和可信度优先尝试：

1. **失败测试**：unit / integration / e2e，落在能触发真实 bug 模式的位置。
2. **HTTP 脚本**：`curl` 或小脚本打本地 dev server，断言状态码、响应体或日志。
3. **CLI fixture**：固定输入，diff stdout / stderr / 输出文件。
4. **Headless browser**：Playwright / Puppeteer 驱动 UI，断言 DOM、console、network。
5. **Trace replay**：保存真实请求、payload、事件日志，在隔离路径中重放。
6. **Throwaway harness**：只启动触发 bug 所需的最小系统子集。
7. **Property / fuzz loop**：对“偶发错误输出”跑多组输入，提高命中率。
8. **Bisection harness**：对 commit、数据集、版本差异自动跑 `git bisect run`。
9. **Differential loop**：同一输入对比旧版本 / 新版本、配置 A / 配置 B。
10. **HITL 脚本**：人必须参与时，用脚本驱动人收集固定输出。

反馈环本身也要迭代：

- 更快：缓存 setup，缩小测试范围，跳过无关初始化。
- 更准：断言具体症状，不只断言“没崩”。
- 更稳定：固定时间、随机种子、文件系统、网络依赖。

非确定性 bug 的目标不是一次干净复现，而是提高复现率。可以循环触发、并行触发、加压力、缩小时序窗口，直到失败率足够支撑定位。

实在无法建立反馈环时，停止并列出已尝试路径；向用户请求可复现环境、HAR / 日志 / core dump / 带时间戳录屏，或请求临时生产插桩许可。

## 3. 假设与分析

列出 3-5 个合理假设，按可能性排序。每个假设必须可证伪，写清预测：

> 如果 X 是根因，那么改变 / 观察 Y 会让症状消失、恶化或出现特定信号。

| 层级 | 检查方向 |
|------|---------|
| 语法/类型 | 拼写、类型不匹配、import 错误 |
| 逻辑 | 条件反转、边界未处理、状态错误 |
| 集成 | 接口不匹配、数据格式变化 |
| 环境 | 配置、依赖版本、权限 |
| 并发 | 竞态、死锁、时序依赖 |
| 数据 | 脏数据、空值、编码问题 |

### 复杂 bug → 派发只读子任务隔离分析

当假设超过 3 个或涉及大量代码探索时，派发**只读子任务**（subagent / parallel agent / sub-call，按当前 agent 平台能力命名）做隔离分析：

- 将收集的 bug 信息作为 prompt 传给子任务
- 子任务在独立 context 中做假设排查（grep、读代码、git bisect）
- 子任务返回根因分析报告
- 主流程基于报告执行修复
- 跨 agent 适配：有子任务能力的平台按其原生命名派发；无子任务能力的平台降级为主流程顺序排查

## 4. 定位

逐个验证假设，每个假设必须有证据支撑或排除：

- **二分法**：git bisect / 注释代码 / 日志注入
- **最小复现**：从完整场景逐步剥离到最小触发条件
- **定向插桩**：每个 probe 对应一个假设预测，一次只改变一个变量
- 确认根因后才进入下一步

### 调试插桩规则

- 优先 debugger / REPL；其次是边界处的定向日志；每条日志服务一个假设预测。
- 临时日志必须带唯一前缀，例如 `[DEBUG-a4f2]`。
- 收尾必须搜索该前缀并清理所有临时插桩。
- 性能问题先建立 baseline measurement，再二分或 profile，用测量结果判断根因。

## 5. 根因 checkpoint

进入修复前必须形成根因 checkpoint，包含：

- 根因是什么（一句话）
- 证据（file:line、日志、复现输出）
- 修复方案（怎么改、改哪里）

当存在高风险改动、权限边界、数据迁移、线上影响，或用户需求与代码事实冲突时，必须用 AskUser 确认后再动代码。普通明确 bug 修复不阻塞等待，但要把 checkpoint 写入过程记录，并继续走失败测试 → 最小修复。

## 6. 修复

1. 先写捕获 bug 的失败测试
2. 最小修复（只改必要的，不顺手重构）
3. 跑通测试
4. 重跑第 2 步的原始反馈环

## 7. 验证

- 原始 bug 已修复
- 回归测试通过
- 边界用例检查
- 临时 debug prefix / throwaway harness 已清理
- 若缺少正确测试位置，把“无可锁定回归的结构原因”记录为后续重构输入

## 2-strike Handoff（硬规）

连续 2 次假设验证失败 → 强制停止当前路径，输出 Handoff 报告，并切换到 `/think-unstuck`。

进入 Handoff 后，当前路径停止追加变体或参数试跑。Handoff 用以下 6 个固定表头：

```markdown
## Handoff

### Symptom
症状一句话描述 + 可复现命令（精确到参数）。

### Hypotheses Tested
| # | 假设 | 验证手段 | 失败原因 |
|---|------|---------|---------|

### Evidence Collected
已经掌握的事实数据：错误信息原文、日志片段、file:line、git blame 结论、版本/环境快照。

### Ruled Out
| 方向 | 排除证据 |
|------|---------|

### Unknowns
仍不清楚的关键点（用问句列出，每条独立一行）。

### Suggested Next Steps
至少 2 个候选路径，每条说明：动作、预期信号、若失败的退路。
```

Handoff 输出后显式切到 `/think-unstuck`，由新流程接管未解问题。

## Bug 修复前置条件（硬规）

报告 `fixed` 之前，必须满足前两条；如果本轮要 commit，再满足第 3 条：

1. **存在失败测试**：能在旧代码上失败（red），新代码上通过（green）。缺少失败测试时，当前状态是 `partial: no regression lock`。
2. **测试落在项目测试套件**：放在项目正式测试目录（`tests/`、`__tests__/`、`*_test.go` 等），不放临时文件、scratch、调试脚本。后续回归靠它兜底。
3. **commit message 或 final 写明覆盖缺口**：如果用户明确要求 commit，commit message 必须解释"为什么这个 bug 之前没被测试覆盖到"；未 commit 时，在 final 的验证/覆盖缺口中说明。

前两条均满足，且覆盖缺口已在 commit message 或 final 中说明时，报告状态才能写为 `fixed`；否则写为 `partial` 并列出缺口。

## Gotchas

- 先找最小触发条件，再进入修复
- 假设不是结论，所有归因都要有证据
- 一次只改一个定位变量，保留根因可追踪性
- 调试修复保持单一目的；重构作为独立步骤处理
- 临时日志没有唯一 prefix，后续通常会漏删

## Required debugging behavior

- 修复基于可复现反馈环和根因 checkpoint。
- 每次修改对应一个假设，并记录预期信号。
- 多处修改需要拆成可验证的小步；如果已经混在一起，先回到可观察状态，再逐个验证。
- 重构作为独立步骤处理；bug 修复阶段只做必要改动。

## 关联技能

- 修复完成后 → `/guard-verify` 验证修复有效且无回归
- 需要大范围探索代码 → 派发只读子任务（按 agent 平台命名）
- 修复后如需重构 → `/dev-refactor`
- 连续失败 2 次 → `/think-unstuck`
