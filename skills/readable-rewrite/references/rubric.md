# Readable Rewrite Rubric

writer 和 critic 共同遵循的评分标准。任何一方对照本文件给出 finding 或修改决策；不允许任一方自带"私有标准"。

## 1. 五槽位定义

每个被讲透的对象必须填这 5 个槽位。空槽位标 `[待澄清: 缺什么]`，不许编造。

| 槽位 | 写什么 | 不接受 |
|------|--------|--------|
| 是什么 | 一句话定位（功能 + 边界） | "用于处理 X" 这种性质描述 |
| 机制 | 下列任一显式形式：伪代码 / I/O 示例 / mermaid 流程图 / mermaid 状态图 / mermaid 序列图 / 决策表 | 仅自然语言描述判断逻辑 |
| 输入 → 输出 | 调用面、参数、返回 | "接收数据并处理" 这种空话 |
| 触发时机 | 谁调用、什么时候、什么事件 | "在合适的时候" |
| 边界属性 | { 完整性, 固定/动态, 来源, 清理路径 } | 默默省略其中任意一项 |

边界属性四子项：

- **完整性**：本节列的规则是全部还是举例（如 6 条命中规则要列全 6 条，不能"举一例代表"）
- **固定/动态**：硬编码 / 配置驱动 / 运行时判定
- **来源**：代码路径 `file:line`、用户输入、外部状态
- **清理路径**：注入的状态是否需要清理、何时清理（无清理需求也要显式标"不适用"）

## 1.1 机制形式选型

逻辑特征对应推荐的显式形式（任选其一，不必同时用多种）：

| 逻辑特征 | 推荐形式 | 反例 |
|---------|---------|-----|
| 顺序步骤 + 少量分支 | 伪代码 | 别把 3 行能讲完的画成大流程图 |
| 多条件判定（>= 3 条件） | 决策表 | 别用 if/else 嵌套伪代码 |
| 多状态转换 | mermaid stateDiagram | 别用文字列举每对转换 |
| 多组件协作时序 | mermaid sequenceDiagram | 别用"先 A 再 B 再 C"的散文 |
| 数据形态变换 | I/O 示例（before / after） | 别只说"会被处理成 X" |
| 流程含循环或早返 | mermaid flowchart | 分支多用图，分支少用伪代码 |

**原则**：一个对象一种形式，不允许伪代码 + 图叠加。3-5 行能讲完优先伪代码；分支或状态多到看伪代码累的，再升级到图。

## 2. 评分维度（四类）

### 2.1 structure（结构）

| 信号 | 级别 |
|------|------|
| 多对象同维度横向并列作为主结构（"脚本 × 维度" 矩阵） | P0 |
| 节内主题混杂（一段讲两件事） | P1 |
| 引用未定义的对象（先用后讲） | P1 |
| 节序颠倒（细节先于地图、例外先于主线） | P2 |

**纯属性对照表例外**：单一维度比较（如"runtime 支持矩阵"、"平台 ✓/✗ 表"）是合法的辅助，不算 P0。判定标准：

- 表格的行是否都属于同一类对象（如都是 runtime 名）
- 表格的列是否仅展示同一种属性（如都是"是否支持 X"）
- 不能既是"多对象"又是"多维度"

### 2.2 slot（槽位完整性）

| 信号 | 级别 |
|------|------|
| 机制槽位无任一显式形式（伪代码 / I/O 示例 / mermaid 图 / 决策表） | P0 |
| 边界属性四子项缺任意一项 | P0 |
| 是什么 / 输入输出 / 触发时机 任一槽位空 | P1 |
| 伪代码存在但关键判定逻辑用自然语言糊过 | P1 |

### 2.3 fidelity（保真）

| 信号 | 级别 |
|------|------|
| 改变原文事实、数字、单位、范围 | P0 |
| 改变原文因果关系或限制条件 | P0 |
| 抹掉原文的例外、风险提示 | P0 |
| 引入原文没有的判断、承诺、推论 | P0 |
| 引入原文没有的具体识别符（命令名、文件路径、数字、特定术语） | P0 |
| 改变量词或对照（"多/强/必须/可能"、"SSOT/举例"、"等/穷尽"） | P0 |
| 术语替换造成语义偏移（如"数据证据" → "数据准确性"） | P0 |

fidelity 类一律 P0，不接受 P1/P2。

**Slot-aware fidelity（避免概括槽位误判）**：

- **「是什么」slot**：一句话定位概括，允许 paraphrase。只有当 paraphrase 引入原文未出现的具体识别符（命令名、文件路径、数字、特定术语）或改变 scope/category 时，才判 fidelity 违规
- **「机制」「输入→输出」「触发时机」「边界属性」slot**：事实陈述。paraphrase 必须保留原始字面信息（数字、术语、命令名、量词、对照关系）

**例**：
- ✅ 「是什么」写"长任务、批处理、dry-run / apply 任务的契约提醒 capsule"——原文涉及 migration / sync / backfill / batch / dry-run / apply，paraphrase 合规
- ❌ 「输入→输出」写"要求 `/guard-secure` 介入"——原文只说"要求安全审查"，引入了原文没有的 slash 命令名
- ❌ 「输入→输出」写"dry-run 数据准确性"——原文是"dry-run 数据证据"，"证据"和"准确性"语义不同

### 2.4 ai-flavor（AI 味）

| 信号 | 级别 |
|------|------|
| em dash 过载（每段 >= 2 个） | P2 |
| rule of three（"不是 X 不是 Y 而是 Z"句式连续 >= 2 段） | P2 |
| AI 词汇命中（delve / leverage / navigate / unlock / 赋能 / 打造 / 重塑 / 助力 / 生态） | P2 |
| negative parallelism（"不仅是...更是..."套娃 >= 2 处） | P2 |
| 千篇一律句长（连续 >= 5 句长度接近、节奏单调） | P2 |
| 空泛承诺（prevent / ensure / guarantee / eliminate / seamlessly 无证据） | P1 |
| 通用 "-ing" 转折开头（Building on / Looking at / 随着...的不断深化） | P2 |
| 元叙述堆叠（"本文将带你深入了解..."） | P2 |
| 滥用 mermaid（3 节点流程图、装饰性图、画图代替思考） | P2 |

## 3. Hard rules（不可违反，违反即 P0）

1. 先输出对象清单，再输出任何重写正文
2. 每对象填满 5 槽位（空槽位标 `[待澄清]`，不许编造）
3. 机制槽位含伪代码或 I/O 示例（非自然语言）
4. 禁止把多对象塞进同一张横向对照表作为主结构
5. 按依赖顺序排对象
6. 一对象一节讲透再下一个，不允许穿插

## 4. Finding ID 规则

格式：`<dimension>-<section-anchor>-<slot>-<keyword>`

- `dimension`: `structure` / `slot` / `fidelity` / `ai-flavor`
- `section-anchor`: 定位锚（节标题简写、对象编号、或行号区间）
- `slot`: `identity` / `mechanism` / `io` / `timing` / `boundary` / `overall`（不属于单一 slot 时用 `overall`）
- `keyword`: 1-3 字问题关键词，必须能区分同一对象同一 slot 内的不同 issue

例：

- `structure-section2-overall-横向铺表`
- `slot-对象3-mechanism-无伪代码`
- `fidelity-对象3-io-数据证据`
- `fidelity-对象3-boundary-等词`（与上行 ID 不同，因为 slot 不同）
- `ai-flavor-section1-overall-emdash`

**ID 稳定性硬约束**：

- 同一 (对象, slot, issue) 三元组在不同轮里必须用同一 ID
- 不同 issue 即使 dimension + section + slot 都相同，keyword 必须区分（如 `数据证据` vs `等词` 是同一对象同一 slot 内的两个独立 issue）
- 不允许跨 issue 复用 ID

orchestrator 用 ID 集合算 Jaccard 相似度判断震荡；ID 不稳定会破坏震荡保护。
