# Scope

与 think-plan 区别：think-scope 解决"agent 和用户对系统现状和改动意图的理解是否一致"，think-plan 解决"方案怎么设计、怎么拆步骤"。scope 清晰后再进 plan，不要跳过对齐直接写 spec。

## Decision Principles

- think-scope 优化的是**需求理解的准确性**，不是需求的完备性或方案设计。
- 返工的主因是 agent 对系统现状的心智模型与现实不符，而不是缺少 spec。
- 代码是需求对齐的锚点：先读代码建立系统上下文，再用上下文去理解需求，比纯对话式追问更有效。
- 当"要改什么、不改什么、为什么改"都明确时停止，不要继续扩展成完整 spec。

## 核心循环

为什么这样循环：返工的主因是 agent 在没有读代码的情况下就开始理解需求，建立了错误的系统心智模型。Ground → Model → Align 的顺序强制 agent 先用代码证据建立上下文，再基于上下文理解需求、提出知情的问题，避免纯对话式追问中因缺乏系统知识而问出无关或误导性的问题。

```
1. Ground  — 读代码，建立系统现状模型
2. Model   — 陈述"我理解的现状 + 我理解的你要什么"
3. Align   — 交互式对齐（代码知情的提问）
4. Scope   — 产出锚定后的 scope
```

### Phase 1: Ground（代码分析）

基于 `$ARGUMENTS` 中的需求描述，定向分析代码：

1. 识别需求涉及的功能区域（入口文件、核心模块、数据流）
2. 读关键文件，理解当前实现的结构、约定和约束
3. 找到与需求相关的现有行为、边界条件、测试覆盖
4. 识别领域术语：代码中的命名、注释、文档是否与用户用词一致

分析范围聚焦在需求相关区域，不做仓库级全扫描（那是 think-map 的职责）。

使用工具而非猜测：Glob/Grep/Read 定位文件，不凭经验推断。大型项目可派发只读子任务并行分析不同模块。

### Phase 2: Model（陈述理解）

向用户陈述两件事：

**系统现状**（基于代码证据）：
- 当前系统在这个区域是怎么工作的（关键路径、数据流、约束）
- 涉及哪些文件/模块/接口
- 现有行为中哪些与需求直接相关

**需求解读**（基于需求描述 + 代码上下文）：
- 我理解你要做的改动是什么
- 改动会影响哪些现有行为
- 我的理解中有哪些不确定点

陈述必须带 `file:line` 或至少具体文件路径作为证据。未确认的推断标 `[推断]`。

### Phase 3: Align（交互式对齐）

基于 Phase 2 的陈述，与用户交互对齐。继承 think-refine 的提问纪律：

- 一次一个问题，不要一次列 5 个
- 每个问题提供 2-3 个候选选项 + 推荐，选项基于代码现实而非凭空构造
- 不替用户做决定

#### Ambiguity Score（停止条件）

四维评分，每维 0-1，越低越清晰：

| 维度 | 含义 | 为什么 |
|---|---|---|
| System Model | agent 对系统现状的理解是否经过验证 | 防止基于错误心智模型做决策 |
| Intent Clarity | 用户要改什么、为什么改是否明确 | 防止解决错误问题 |
| Boundary Clarity | 改什么 / 不改什么是否明确 | 防止 scope creep |
| Acceptance Clarity | 怎么算改好了 | 防止无法验收 |

总分 = 四维平均。 <= 0.20 放行，> 0.20 继续追问。

评分是 agent 自检工具，不是展示给用户的流程。哪维高就追问哪维。

#### 代码知情提问示例

| 纯对话式提问 | 代码知情提问 |
|---|---|
| "你要加什么类型的认证？" | "当前 `auth/middleware.ts:23` 用 JWT + cookie session。你是要在这套机制旁加 OAuth provider，还是替换掉？" |
| "数据存哪里？" | "`models/user.py:45` 的 User 表已有 email/password 字段，`services/cache.py` 用 Redis 做 session。新功能的数据要加字段还是新建表？" |
| "需要考虑什么边界情况？" | "`handlers/upload.ts:78` 当前限制 10MB，`config/limits.json` 可配置。你的需求是否需要调整这个限制？" |

#### 术语对齐

代码分析中发现的命名与用户用词不一致时，主动提出：

- "你说的'订单'在代码里叫 `Transaction`（`models/transaction.py`），后续用哪个词？"
- "代码里 `publish` 和 `release` 是两个不同操作（`services/publish.py` vs `services/release.py`），你说的'发布'指哪个？"

### Phase 4: Scope（产出）

对齐完成后产出 scope 文档（直接输出到对话中，不创建文件）：

```markdown
## Scope

### 系统现状
[基于代码的关键路径描述，带 file:line 证据]

### 改动意图
[一句话：要做什么 + 为什么]

### 改动范围
| 区域 | 现状 | 目标状态 | 涉及文件 |
|---|---|---|---|
| ... | ... | ... | ... |

### 不改什么（显式排除）
- ...

### 验收标准
- [可执行的验收条件]

### 待决策（如有）
- [对齐过程中未解决的问题]

### 下一步
- 任务简单 → 直接动手 / `/dev-tdd`
- 任务复杂 → `/think-plan` 产出完整 spec
```

## Evidence / Stop / Risk

**Evidence**：scope 中的系统现状描述必须有 file:line 证据支撑，不接受纯推断。

**Stop**：Ambiguity Score <= 0.20 时停止追问。3-5 轮无法收敛时，给出 2-3 个候选 scope 让用户选择，附各自取舍。

**Risk**：
- 代码分析阶段发现需求可能触碰高风险区域（数据迁移、外部 API、权限变更）时，在 Model 阶段显式标出
- 发现需求与现有代码契约冲突时，放入待决策而非沉默选边

## 跳过条件

以下场景不需要本 skill：

- 需求已经具体到文件级 + 验收方式明确（直接动手）
- 典型 bug 修复，复现路径清晰（用 `/dev-debug`）
- 纯机械操作：改名、格式化、rebase
- 仓库级全局了解（用 `/think-map`）

## Gotchas

- 不要把 Ground 阶段做成 think-map 级别的全仓库扫描；只读需求相关的代码
- 不要跳过 Ground 直接问用户——这和 think-refine 没区别，失去了代码锚定的价值
- 不要把 scope 写成 spec——scope 回答"改什么/不改什么"，spec 回答"怎么改"
- 3-5 轮追问无法收敛时给候选方案让用户选，不做无限循环
- 用户说"你看着办"时给推荐 + 兜底，不真的随便办

## 关联技能

- 仓库全局了解 → `/think-map`
- scope 确定后需要完整方案 → `/think-plan`
- scope 确定后任务简单可直接开发 → `/dev-tdd`
- 技术不确定需要调研 → `/think-research`
- 连续失败 2 次 → `/think-unstuck`
