## A. Memory 真实实践横向(校准理论)

三项目 memory 横向提炼（喂正在做的 memory 算法选型）。约束：文件型、不要向量/图DB、跨 5 agent 软链共享。

A. 被三项目一致验证、高置信度该抄（交集，confirmed）：

1) 正文用文件/markdown、调度计量用 SQLite、职责分离。hermes(MEMORY.md/USER.md 纯 md + state.db FTS5)、openclaw(MEMORY.md + JSON 状态 + jsonl digest)、codex(~/.codex/memories 是 git markdown，SQLite 只存 stage1_outputs/jobs 不存正文)。最强信号：正文必须人可读、可手改、可 git 化；DB 若有只协调/计量。codex 直接把 memory_root 做成 git repo——而 ~/.dotfiles 本身就是 git 仓库，零额外基础设施，天然契合。

2) 两段式巩固：廉价确定性 + 昂贵可选 LLM。三项目惊人一致且独立验证：hermes(curator 纯函数老化 active→stale→archived 默认开 / LLM umbrella 合并默认关)、openclaw(dreaming light→REM→deep，REM 用正则分类 durable vs situational 无 LLM)、codex(Phase1 廉价模型抽取 / Phase2 medium 模型全局合并解耦)。与 fragility-spike「先廉价验证再实施」纪律同构。最该抄：先上纯函数/正则的入库预筛+老化，LLM 合并作人工/定时可选 pass。

3) char/token 预算强制遗忘 + 模型无关口径。hermes(用 char limit 不是 token limit，理由 char count 模型无关) + openclaw(MEMORY.md 10000 char budget，刻意低于 bootstrap 注入上限) + codex(注入 summary 设硬 token 上限截断)。我们跨 5 agent 共享同一文本，char-count 模型无关口径正中需求。配套：丢弃要返回被丢标识进日志，绝不静默(openclaw log-and-continue：丢老保新、拒绝吞最新材料)。

4) 遗忘=降权+标记可见+容量裁剪，几乎不硬删。hermes(archive 非 delete + restore + pin 保护)、openclaw(时间衰减不删 / contested-superseded 标记 / stale dashboard)、codex(git-diff 语义遗忘：删支撑文件→diff→合并 agent 清引用)。codex 的「git diff 当统一遗忘+增量合并引擎、无变化直接 succeed」对我们 ROI 最高(已有 git + SSOT 红线)。

5) salience 由强 prompt schema 承载 + 显式反模式黑名单。hermes(memory schema 的 WHEN/SKIP + Do-NOT-capture) + codex(No-op gate「未来 agent 会因此更好吗」+ evidence→implication 句式 + redact_secrets) + openclaw(REM durable/situational 正则)。

B. 与理论一致/有认知科学依据：openclaw 6 信号加权提升评分(relevance/frequency/diversity/recency/consolidation/conceptual)，frequency 用 log1p 防刷分、consolidation 奖励跨天反复=spacing effect 间隔重复；hermes asymmetric trust(罚 0.10 > 奖 0.05)。两者都把「什么值得长期记」量化成可解释可日志公式而非 LLM 黑盒——这是我们目前 memory 全等权重的直接升级路径。

C. 与约束冲突/不抄：hermes HRR 全息向量代数(单用户 over-engineering)、所有第三方 SaaS provider wrapper(honcho/mem0/supermemory/byterover)、openclaw LanceDB/sqlite-vec 向量后端(违反「不要向量DB」硬约束)。三项目都独立证明：检索可零向量零 LLM——hermes session_search 纯 FTS5 bookend+window、openclaw memory-core 纯 lexical+JSON 评分、codex grep MEMORY.md 渐进披露。这是我们「不要向量DB」约束的有力背书。

D. 我们文件型跨 agent 共享必抄的安全/并发护栏（三项目都有等价物）：
- external-drift 检测+拒写+.bak 备份(hermes memory_tool.py:647-700)：每次写前 re-read，round-trip 不一致就判被外部改过、拒写、备份、回 remediation，绝不静默覆盖。多 agent 并发写同一文件是默认场景，几乎必抄。
- atomic temp+rename+fsync + flock 锁(.lock 而非锁正文，hermes) / 进程内锁+文件锁含 PID 存活检测(openclaw isProcessLikelyAlive) / SQLite jobs 分布式锁 lease+ownership_token(codex)。直接搬给 .long-loop vault + 未来跨 agent memory 文件。
- memory 进 prompt 前过 threat-pattern 扫描，命中替换 [BLOCKED] 进 prompt 但原文留盘(hermes)。跨 agent memory 注入面更大，我们 capsule 注入链目前没有这道。
- 内容指纹 sha256 + ledger 幂等防重导入(codex external-agent-sessions)。跨 agent raw memory 收集时每条带指纹、合并端 ledger 去重，避免同一会话被多 agent 重复记——直接回应 memory 备忘「machine-local + CC-only」局限。

E. 跨 agent 写入范式（最该照搬的工作流）：codex 两阶段 Phase1(单会话抽取，per-rollout 可并行 concurrency=8)+Phase2(全局串行抢锁合并)，落地形态 = Phase1 在 stop_check hook 产出结构化 raw memory 写 ~/.dotfiles/raw_memories/(带 No-op gate/redact/evidence→implication)，Phase2 = 手动/定时 /assist-consolidate 全局合并成 MEMORY.md。stage_one_system.md + consolidation.md 两份 prompt 几乎可直接改写成我们 skill 指令。hermes 的「nudge 触发→后台 fork review→不抢主对话注意力」是这套写入的理想触发范式，我们已有 background subagent 编排经验(dev-long-run tmux pane)。

F. 三分法/结构（字段设计参照）：hermes 强制区分 MEMORY(agent 笔记:环境/约定/工具怪癖/教训) vs USER(用户画像) vs SKILL(如何做某类任务)，「可复用流程进 skill 不进 memory」；codex MEMORY.md 块头自带 applies_to:cwd=... + reuse_rule= 做 cwd 边界路由 + v1 schema 版本头(首行不符整体重建)。这印证我们 capsule(纪律)/skill(能力)/memory(状态)分层，且 umbrella 合并判据是「人类维护者会写成几个」而非相似度——直接回应 harness-governance 真问题(近邻区分度+发散缺口)。

## 不要抄(anti-patterns)

- 向量DB/HRR 全息向量代数(hermes holographic plugin)：单用户 agent over-engineering，且违反我们「不要向量DB」硬约束。三项目都独立证明检索可零向量零 LLM(FTS5 bookend / lexical+JSON 评分 / grep 渐进披露)——别被「记忆系统必须语义检索」的理论带偏。
- 第三方 memory SaaS provider wrapper(hermes 的 honcho/mem0/supermemory/byterover/hindsight 等)：对我们参考价值低，引入外部依赖违反零外部依赖/本地化原则。
- 靠 git log/summary/记忆事后推断 skill 去向或合并关系(hermes absorbed_into 显式声明 > 事后推断三信号 reconcile 正是打脸这条)：自动治理必须靠显式 provenance 声明，不靠脆弱的事后启发式推断。
- 把环境依赖失败/对工具的负面断言('browser 工具不能用'/'X 坏了')/已解决的瞬时错误/一次性任务叙事写成持久 memory 或 skill(hermes background_review.py:249-268)：'these harden into refusals the agent cites against itself for months after the actual problem was fixed'。跨 agent 共享时一条毒化全部 5 agent。记 FIX(install/config/env)而非'此工具坏了'。
- session-keyed/缓存的 skill/tool catalog(hermes tool_search docstring 引 openclaw#84141 cron 回归教训)：catalog 必须无状态实时重建，否则与实时注册表 drift 导致静默丢工具。我们若自建 skill 索引必须避免。
- 把渐进暴露当成解决「技能多」的银弹(hermes tool_search 阈值门控只在占 context >10% 才启用)：渐进暴露解决的是 token 占用不是误选；我们 MEMORY 已校准真问题是近邻区分度+发散缺口(harness-governance)，别再误投渐进暴露。
- 硬删记忆 + last-write-wins 覆盖：三项目都选 archive/标记/容量裁剪可逆可审计 + drift-detect-then-refuse，多写者并发文件存储用 last-write-wins 必然互相覆盖丢数据。
- 定期全量重扫判过期做遗忘(贵且易误删)：codex 把遗忘降维成 git diff 处理(删支撑文件即遗忘信号、无变化直接 succeed)，比全量重判廉价且精确。
- 为小工具集/小规模付重型机制复杂度：hermes 阈值门控、openclaw circuit breaker 都有「不到阈值就 no-op 直通」的克制。非对称 override 矩阵、完整三阶段 dreaming 对我们当前规模偏重，先借单阶段+评分。
- 引入 CC-only 特性默认假设跨 agent 平价(违反 CC-native-feature-adoption-policy)：codex 的运行期 skill 暴露门禁、各 agent skill 加载机制不同，部分能力只能做分发期 preflight 做不到运行期统一隐藏，必须显式记录不可用而非默认平价。

## 各项目 memory 原始发现(节选)

### hermes-agent (Nous Research). 自称"唯一内置 learning loop 的 agent"。其 memory 系统是两层:(1) 

- **storage**:三套存储介质并存,按用途分层(confirmed): (1) 永久 curated memory = 纯 Markdown 文件,MEMORY.md(agent 自己的笔记)+ USER.md(用户画像),§ 分隔 entry,profile-scoped 在 ~/.hermes/memories/(tools/memory_tool.py:55-59,113-130)。(2) 历史对话 = SQLite state.db + FTS5 虚拟表,带 session lineage(parent_session_id 链)(session_search_tool.py:68-87,402-413)。(3) 结构化 fact = 独立 memory_store.db,facts/entities/fact_entities 三表 + FTS5 + memory_banks(HRR 向量 BLOB)(holographic/store.py:16-76)。三者都用 SQLite/文件本地化,零外部依赖。关键工程点对我们的价值: a) memory_tool 用"char limit 不是 token limit"做预算,理由是 char count 模型无关(memory_tool.py:18-24,124-128 注释)——我们跨 5 agent 共享同一 capsule/纪律文本,正好需要模型无关的预算口径。b) 所有写盘走 atomic temp-file + os.replace + fsync,先写 .lock 文件做 flock 而不锁正文,避免 truncate 竞态(memory_tool.py:207-242,702-731)——我们 .long-loop vault 和 memory 文件多 agent 并发写时直接可抄。c) external-drift 检测:每次写前 re-read,若磁盘内容 round-trip 不一致或单 entry 超限就判定"被 patch 工具/shell/另一 session 改过",拒绝写并备份 .bak,绝不静默覆盖(memory_tool.py:647-700,issue #26045)——这正是文件型跨 agent 共享最大的数据丢失风险,我们必须有同款守卫。
- **retrieval**:检索完全无 LLM、无向量 DB 也能跑(confirmed)。(1) 历史对话 session_search 是单一 tool 四种 shape(discovery/scroll/read/browse),靠传参推断 mode 无显式 mode 参数(session_search_tool.py:495-616)。discovery 的精妙处:FTS5 命中后按 session lineage 去重(沿 parent 链 resolve 到根,避免同一对话的 fork/compaction 子 session 重复占榜,session_search_tool.py:428-447),每个命中返回三段: bookend_start(会话头 3 条=目标/起点)+ ±5 条命中窗口(锚点标记)+ bookend_end(尾 3 条=结论/决策),让模型零成本重建"目标→命中→结论"而不付全 transcript 代价(session_search_tool.py:450-483, schema 646-660)。scroll 用 around_message_id 翻页,边界消息在前后窗口都出现作为方向标。(2) fact store 是 hybrid 三路加权: FTS5(0.4)+ Jaccard token 重叠(0.3)+ HRR 余弦(0.3),再乘 trust_score,可选 temporal decay 0.5^(age/half_life);numpy 不在时自动把 HRR 权重重分配给前两路、纯关键词降级(retrieval.py:38-112)。对我们的价值: bookend+window 的"廉价上下文重建"模式可直接套在我们要建的跨 agent memory 检索上(纯 SQLite FTS5,零向量 DB,符合用户硬约束);lineage 去重提醒我们跨 agent 时也要按"任务/会话根"去重而非按物理文件。HRR 那套全息代数对单用户 agent 是 over-engineering,我们不需要。
- **encoding**:编码/salience 判定有两条独立链路(confirmed)。(1) 写入门槛由 tool schema 的自然语言指令承载,而非代码硬规则: memory schema 明确"WHEN: 用户表达偏好/纠正/个人细节,或学到稳定环境/约定/工作流事实就主动存;优先级 用户偏好&纠正 > 环境事实 > 流程;SKIP: 琐碎/易重新发现/原始数据/任务进度/已完成日志/临时 TODO;可复用流程应进 skill 而非 memory"(memory_tool.py:1013-1023)。这把"什么值得记"的 salience 完全交给模型判断,但用强 prompt 收敛。(2) fact store 的 salience 是显式 trust_score: 新 fact 默认 0.5,asymmetric 反馈 helpful +0.05 / unhelpful -0.10(惩罚重于奖励,store.py:78-82,353-392),retrieval_count/helpful_count 计数,检索按 trust 过滤+排序——这是个可量化的"经验衰减/强化"机制。还有写入安全:所有进 system prompt 的 memory 内容在 snapshot 构建时跑 threat-pattern 扫描(strict scope),命中就替换成 [BLOCKED] 占位符进 prompt 但原文留在磁盘给用户删,防止 poisoned memory 注入(memory_tool.py:78-80,171-205)。对我们: schema 即规范的做法和我们 capsule 注入纪律同构;asymmetric trust(罚>奖)是个轻量可抄的"经验可信度"标量,比我们现在 memory 全等权重更稳;memory 进 prompt 前过 threat 扫描这一条,我们 context_capsule 注入链也该补(跨 agent memory 会被多 agent 读,注入面更大)。
- **consolidation**:巩固是这套系统最有借鉴价值的部分,分两个时间尺度(confirmed)。(1) 即时/在线巩固——自我 nudge: 按 user-turn 计数(_memory_nudge_interval 默认 10)和 tool-iteration 计数(_skill_nudge_interval 默认 10)各自触发(turn_context.py:252-260, turn_finalizer.py:409-415, agent_init.py:1142,1235)。触发后不是阻塞当前对话,而是在响应已交付给用户之后 fork 一个 daemon 线程跑 review(turn_finalizer.py:425-435, background_review.py:4-10,839),"never competes with the user's task for model attention"。fork 继承父 agent 的 model/credentials/cached system prompt 以命中同一 prefix cache(实测省 ~26% 成本, background_review.py:678-687),skip_memory=True 防止 review 把 harness prompt 污染进真实 memory namespace(background_review.py:623-637)。review 提示词把会话回看一遍,决定是否 memory.add 或 patch/create skill。(2) 离线/批量巩固——curator: inactivity-triggered(无 cron daemon),idle 且距上次 >interval(默认 7 天)就跑(curator.py:219-269)。两段: 先跑纯函数 deterministic 老化(active→stale→archived 按 last_activity 时间戳,无 LLM, curator.py:276-331),再可选(默认关,省 aux 成本)跑 LLM "umbrella-building" 把一堆 narrow skill 合并成 class-level umbrella(curator.py:365-504)。结果分类用三信号 reconcile: 模型在 delete 时声明的 absorbed_into(权威)> 模型事后 YAML 块 > tool-call substring 启发式审计(curator.py:808-936)。对我们价值极高: a) "nudge 触发→后台 fork review→不阻塞主对话"是我们想建的跨 agent memory 写入机制的理想范式,且我们已有 background subagent 编排经验(dev-long-run 的 tmux pane);b) "deterministic 老化(纯函数 无 LLM)+ 可选 LLM 合并"两段式,正好对应我们 64 个 skill 的近邻区分度问题——我们可以先上纯函数的"按 last-used 标 stale"再考虑 LLM 合并;c) absorbed_into 显式声明 > 事后推断,这条直接打脸"靠 git log / summary 事后推断 skill 去向"的脆弱做法。
- **forgetting**:遗忘是显式且可恢复的(confirmed)。(1) skill 永不删除,最大破坏动作是 archive(mv 进 ~/.hermes/skills/.archive/),可 hermes curator restore 恢复(curator.py:16-19,319-321,382-384)。pinned skill 完全跳过自动转换(curator.py:300)。(2) curated memory 靠 char limit 强制遗忘: add 超限直接拒绝并回传当前所有 entry + 指令"在同一个 batch 里 remove/replace 腾空间再 add",用 atomic batch(all-or-nothing,只对最终预算检查,中间溢出不算)避免多轮 consolidate-retry 重发整个上下文(memory_tool.py:296-346,449-554)。(3) fact store 靠 trust 衰减 + min_trust 过滤做软遗忘(检索默认 min_trust=0.3,trust 跌破就检索不到但不删,retrieval.py:48-66)。最关键的设计纪律在 skill review 提示词的"Do NOT capture"负面清单: 禁止把环境依赖型失败(缺二进制/未配置凭证/command not found)、对工具的负面断言('browser 工具不能用'/'X 坏了')、已解决的瞬时错误、一次性任务叙事写成 skill——理由原文"these harden into refusals the agent cites against itself for months after the actual problem was fixed"(background_review.py:249-268)。对我们价值: "记录修复办法而非'此工具不可用'"这条反 anti-pattern 纪律,我们 assist-learn / memory 沉淀必须照抄——这正是文件型经验库最容易自我毒化的失败模式;archive 而非 delete + pin 保护 也应是我们 skill 治理(harness-governance program)的默认。char-limit-forces-consolidation 比无限增长的 memory 更适合进 prompt。
- **structure**:结构上有清晰的"记什么进哪儿"三分法(confirmed),且代码强制区分: MEMORY(agent 笔记: 环境/约定/工具怪癖/教训)vs USER(用户画像: 名字/角色/偏好/风格)vs SKILL(如何做某类任务)。提示词反复强调边界:"memory 说'用户是谁、当前运营状态如何';skill 说'如何为这个用户做这类任务';可复用流程进 skill 不进 memory"(memory_tool.py:1019-1023, background_review.py:231-237,318-323)。skill 库的目标形态是 class-level umbrella(一个富 SKILL.md + references/templates/scripts/ 子目录承载 session-specific 细节),而非"一会话一窄 skill"的长平铺列表——curator 和 review 都把"N 个窄 skill 该不该写成 1 个带 N 个小节的 skill"作为合并判据,而非"两两是否重叠"(curator.py:365-399, background_review.py:175-178)。provenance 用 ContextVar 区分 background_review fork 写的(=agent-created,归 curator 管)vs 前台用户让写的(=用户的,curator 绝不碰)(skill_provenance.py:1-78)。对我们价值: a) memory/user/skill 三分 + "可复用流程进 skill"恰好印证我们 capsule(纪律)/ skill(能力)/ memory(状态)的分层,可作为我们 memory 机制的字段设计参照;b) "umbrella 化、按'人类维护者会写成几个'判合并"直接回应 MEMORY.md 记的 harness-governance 真问题(近邻区分度+发散缺口)——合并判据不是相似度而是维护者视角;c) provenance 区分 agent-created vs user-authored 是我们跨 agent skill 共享必须有的概念(谁创建的、能不能被自动治理)。
