## 思路哲学（Why）

这是一个内容创作者（公众号「数字生命卡兹克」）把**自己一个人的写作人格**固化成可加载指令集的项目。核心不是「教 AI 写好文章」，而是「让 AI 用我的口吻、我的禁忌、我的节奏写」。它要解决的真实痛点是：通用大模型写出来的中文长文有强烈「AI 味」——套话、过度结构化、宏大叙事开头、滴水不漏的客观——这种文风在公众号长文场景里会立刻劝退读者，丢掉「活人感」。

几个关键洞察与取舍：

- **风格是有立场的，不是中立的。** README 明确写「这个 skill 是有立场的，它会拒绝写赋能/抓手/闭环、拒绝首先其次、拒绝在当今 AI 快速发展的时代」。它主动把自己定义为「文风生成器」而不是「通用好文笔」，并直接说不适合追求通用好文笔的人。这是对 scope 的硬切。
- **AI 不替你思考，只放大你。** SKILL.md「第二步：明确 AI 的角色边界」明确划分「AI 擅长做的（找证据/找类比/按确定角度扩写/补背景/调结构）」与「做了会暴露的必须人来（第一手观察、核心创意角度、情绪真实表达、数据到人物的同理心）」，并给出「人提供素材+观点+经历+情绪 → AI 扩写 → 人二次改写 → AI 自检 → 人终审」的协作流。它承认风格可生成、洞察不可生成。
- **把「活人感」工程化。** 最大特色是把一个本来非常主观、难量化的目标（「读起来像真人写的」），拆成可执行的硬规则 + 模式库 + 分层自检，借用软件工程的测试金字塔思想（L1 语法 → L2 单元 → L3 集成 → L4 验收）。
- **hv-analysis（横纵分析法）是同一作者风格的「研究报告」变体**：双轴（纵向追时间深度 = 历时分析，横向追同期广度 = 共时分析）+ 交汇洞察，产出 1-3 万字 PDF。它显式声明「从卡兹克文风借鉴哪些、不借鉴哪些」，做了文体适配（研究报告需要小标题、可放松标点禁令、去掉公众号尾部）。

## 特殊技巧（How）

1. **四层自检体系（测试金字塔类比）**（`khazix-writer/SKILL.md` 第四步）：L1 硬性规则（禁用词/禁用标点/结构套话/工具名扫描，零命中才过）→ L2 风格一致性（开头/节奏/口语化/标点二次确认，给是否判断）→ L3 内容质量（观点支撑/知识输出方式/文化升维/对立面同理心/类型专项）→ L4 活人感终审（最主观，通读问「这是有见识的普通人在聊天还是 AI 在输出信息」）。每层有明确通过标准（如 L2 至少 3/4 通过）和修复指引，最后输出固定格式的 `## 质检报告`。这是把主观文风审稿做成可重复 checklist 的范本。

2. **黑名单工程化为可扫描规则。** 不是泛泛说「别写得像 AI」，而是列出可全文搜索的具体串：禁用词（「说白了」「意味着什么」「本质上」「换句话说」「不可否认」「综上所述」），禁用标点（冒号→逗号、破折号→句号、双引号→「」），并给每一项**配好替换方案**。L1-1 直接是「`说白了` → 换成 `坦率的讲`/`其实就是`」这种映射表。

3. **白名单同样具体——推荐口语化词组库。** SKILL.md「推荐口语化词组」分类给出可主动使用的活人感表达（转场/判断/自嘲/情绪/拉近距离/口头禅），并定量约束「全文至少出现 8-10 个不同口语化表达」「断裂句至少 3 次」。把「有 voice」变成可计数指标。

4. **文章原型分类驱动写法与自检。** 先判断属于五种原型之一（调查实验型/产品体验型/现象解读型/工具分享型/方法论分享型），每种写法重心不同，L3-5「文章类型专项检查」按原型做差异化校验（如方法论型必须每节落到可执行动作 + 坦诚学习曲线和失败点）。这是「先分类再套对应规则」的数据驱动设计。

5. **大量命名化的可复用修辞模式。** HKR 质检（Happy/Knowledge/Resonance）、人物画像法、文化升维、句式断裂、回环呼应（契诃夫之枪）、谦逊铺垫法、读者直呼法、层层剥开、英雄之旅叙事弧、反向论证、逐一展示法（借喜剧「升番」逻辑，最弱放前最炸留后）。每个都有名字 + 定义 + 触发时机，便于在 prompt 里直接调用。

6. **正例/反例对照库**（`references/style_examples.md`）：18 类，每类给真实文章片段而非抽象描述；第 12 节「AI 初稿 vs 卡兹克修改对比」直接并列「AI 写的 / 卡兹克改的 / 差异」三栏，让模型对照学习改写方向（如把假设性「有一次」换成「我今天正在搞的」真实场景）。

7. **方法论 reference 与风格 reference 分离**（`content_methodology.md` 管选题来源/选题分类/节奏/创意案例工作法，`style_examples.md` 管文风片段），SKILL.md 末尾用「参考资料」按需指向，控制主指令体积。

8. **触发与边界设计很重。** `description` 里大量列举触发词（写文章/写稿子/续写/扩写/公众号长文…）+ 隐式触发说明（「即使用户只说帮我把这个写成文章」）+ 明确反向边界（不用于小红书/推特，标题摘要用 wechat-title）。hv-analysis 同样列出与 khazix-writer / wechat-title 的分流。这是把路由判断写进 skill 自身。

9. **hv-analysis 的文体适配显式化**（`hv-analysis/SKILL.md`「写作风格」）：用「从卡兹克文风借鉴的元素」「不借鉴的元素」两张清单，明确研究报告要保留节奏/叙事/层层剥开，但克制口语密度、保留小标题、放松标点禁令、去掉公众号尾部。同一人格在不同文体下的迁移做了显式裁剪。

10. **子 Agent 并行检索 + 工具语义约束**（hv-analysis 第一步）：纵向/横向/补充三路并行；要求 prompt「描述目标（获取/调研）不用暗示手段的动词（搜索/爬取），让子 Agent 自主判断获取方式」；信息来源优先级表（一手>二手）；学术对象强制查 arxiv API。这套对调研类写作 skill 直接可借。

## 可借鉴点（for writing-skills）

- **分层自检体系是最值得整套搬过来的资产。** 我们做中文写作 skills 时可以把「文风校验」做成 L1（可机器扫描的黑名单：禁用词/禁用标点/套话）→ L2（风格模式是否命中，给是否判断）→ L3（内容深度/论证）→ L4（整体活人感终审），并强制输出固定格式质检报告。这天然契合我们「验证比生成贵、inner-loop verifier 与 acceptance verifier 分离」的纪律：L1-L2 是 inner-loop，L4 是人格层验收。
- **黑名单/白名单都要配可扫描串 + 替换映射**，而不是自然语言提醒。符合我们「能靠脚本/规则强制的就别只写成提醒」——L1 的禁用词表完全可以落成一个 `verify_*.py` 式扫描器，而非靠模型自觉。
- **「先分类再套规则」的数据驱动结构**：用文章原型/文体表驱动差异化写法与校验，而不是一堆 if 判断散在正文，符合我们「复杂逻辑写成表」的原则。
- **正例/反例对照（尤其 AI 初稿 vs 人改对照三栏）** 比抽象风格描述更能约束模型，可作为我们 style reference 的标准格式。
- **AI 能力边界显式声明**（哪些放心交给 AI、哪些必须人来）值得放进任何「AI 辅助创作」类 skill 的开头，避免模型越界编造第一手经历/情绪。
- **文体迁移用「借鉴/不借鉴」双清单**：如果我们要做一套覆盖公众号/研究报告/小红书的体系，可借鉴 hv-analysis 那种「同一 voice 在不同文体下显式裁剪」的写法，避免规则冲突。
- **触发词 + 反向边界 + 跨 skill 分流** 写进 description，是多 skill 共存时路由清晰的关键，和我们 `scripts/verify_skills.py` 强制触发前缀的思路一致。
- 注意取舍：本项目是**单作者人格**，强绑定个人语料（具体到「磨平信息差」「老阴逼」等签名表达）。我们若做通用中文写作体系，应把「人格层」与「通用文风工程层」解耦——可借鉴它的工程骨架（分层自检/黑白名单/原型表），但人格内容要做成可替换的配置/语料，而非硬编进 SKILL.md。
