Hook 机制速览

更新时间：2026-05-15。目标：用最少心智负担 review 当前 Droid / Claude Code hooks、影响和风险。

最重要结论：当前 hook_stop_check.py 的 systemMessage 是用户可见提示，不是可靠的 agent 可执行反馈通道。Factory 文档说明：Stop hook 若要让 Droid 继续并看到原因，应返回 {"decision":"block","reason":"..."}，或用 exit code 2 stderr。当前实现是 P1 advisory，不是 hard gate。
当前策略
轻阻断 + 强提醒

PreTool boundary gate 默认 advisory；Stop 只提醒，不阻断。

主要收益
减少 silent boundary drift

封装、schema、metric、data source 等场景会被提前或事后提醒。

主要代价
上下文噪音

PostTool scanner 会给 agent 注入 finding；误报会消耗注意力。

4. 哪些输出进入 agent 上下文

输出方式是否进入 agent当前用在哪里review 重点
hookSpecificOutput.additionalContext 是 SessionStart、UserPromptSubmit、PostToolUse scanners 要短、准、低误报，因为会占用上下文并影响行为。
systemMessage 主要给用户 boundary gate advisory、stop check advisory 不能把它当作可靠的 agent 自动修复机制。
decision:block + reason 是 当前 P1 未默认使用 需要 agent 必须继续处理时再启用；会提高阻断感。
exit code 2 + stderr PreTool/Stop 可见 hook_command_guard.py 风格 适合强阻断；错误文案必须告诉 agent 下一步怎么改。
exit code 0 stdout 通常否 普通 hooks 的日志/JSON Factory 文档说除 UserPromptSubmit / SessionStart 外，Droid 不看 exit 0 stdout。
当前 Stop advisory 的含义：你会在 UI 里看到 {"systemMessage": ...}，但这不是 P1 的 hard gate。若希望 agent 必须继续并补写 Boundary decisions:，应把 Stop hook 改为 {"decision":"block","reason":"..."}。

5. 新增 boundary 机制拆解

Boundary capsule

agents/context-capsules/boundary-decision.md

  • 触发：wrapper、integration、schema、metric、data source、prod、context hook 等关键词。
  • 作用：提醒先问、列 caller、写 contract cases、记录 manifest。
  • 风险：关键词宽，可能在普通讨论 hook/schema 时注入。

Boundary gate

scripts/hook_boundary_gate.py

  • 触发：高风险 prompt 后的首次文件写操作。
  • 放行证据：Boundary decisions:、Callers:、Data source:、Metric route: 等。
  • 当前模式：advisory；可用 BOUNDARY_GATE_MODE=block 升级。

Boundary scanner

scripts/scan_boundary_decisions.py

  • 扫描新增 diff：rejection、silent catch、implicit branch、default、retry、metric routing、schema contract。
  • 通过 PostToolUse additionalContext 注入给 agent。
  • 风险：会命中 hook 自身或测试时的合理边界，已排除部分脚本。

Stop manifest check

scripts/hook_stop_check.py

  • 检测 transcript 是否有 boundary scanner finding。
  • 只接受 assistant 自己写的非空 Boundary decisions:。
  • 当前只 systemMessage 提醒，不阻断。

6. 对效率和效果的影响

机制正面影响负面影响建议阈值
Session / prompt capsules 按需给 agent 正确工作流，减少靠记忆。 关键词误触发会污染上下文。 保持 capsule 短；发现频繁误触发就收窄关键词。
PreTool boundary gate 把“先想边界”前移到写代码前。 可能让简单任务多一步。 P1 advisory；连续事故再切 block。
PostTool scanners 即时发现 silent additions、debug residue、operational gaps。 误报会打断实现节奏。 只保留高信号规则；误报先加排除或收窄。
Stop check 收尾前暴露没有验证或没有 manifest 的问题。 当前不保证 agent 自动处理。 如果你希望强制闭环，升级为 Stop block。
Command guard 降低 shell 误用和破坏性命令风险。 可能阻断复杂但合理命令。 保留强阻断；对误报补 allow / 更精确规则。

7. 快速 review checklist

  • 想知道 agent 一定看见什么：找 additionalContext、decision:block reason、exit code 2 stderr。
  • 想知道只是你能看见什么：找 systemMessage、普通 exit 0 stdout、notification。
  • 想减少噪音：先看 CAPSULE_RULES 和 scanner 排除项。
  • 想增强强制力：把 Stop advisory 改为 decision:block，把 PreTool advisory 改为 permission deny/block。
  • 想保持效率：不要把所有规则都放 PostTool scanner；广覆盖放 capsule，硬规则只扫高信号。

9. 推荐下一步

如果目标是“用户能 review 风险”：保持当前 P1 advisory。

如果目标是“agent 必须补救后才能结束”：把 hook_stop_check.py 的 boundary manifest 问题改成 Stop decision:block + reason。这是更强但更打断的 P2。
