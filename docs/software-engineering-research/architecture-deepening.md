# Architecture Deepening

来源：`refs/mattpocock/skills/skills/engineering/improve-codebase-architecture/`。

## 推荐方案

[确认] 用“深模块”判断结构质量：好的模块不是文件多或行数多，而是调用者用少量接口就能获得大量行为，并把变更、bug、知识和验证集中到一处。

## 术语

| 术语 | 含义 | 检查问题 |
|------|------|----------|
| 模块 | 有接口和实现的任意单元：函数、类、包、垂直切片 | 它是否承载一个清晰概念？ |
| 接口 | 调用者必须知道的一切：类型、约束、顺序、错误、配置、性能特征 | 调用者需要知道的事实是否太多？ |
| 实现 | 模块内部代码 | 复杂度是否被藏在接口后面？ |
| 深度 | 接口杠杆：少量接口背后承载大量行为 | 一个调用能完成多少有价值行为？ |
| 接口位置 | 行为可被替换、注入或验证的位置 | 这是业务变化点，还是为测试硬造的间接层？ |
| 适配器 | 满足某个接口位置的具体实现 | 是否有生产和测试两种真实实现？ |
| 局部性 | 变更、bug、知识、验证集中在一处 | 修一次是否覆盖所有调用点？ |

## Deletion test

假设删除一个模块：

- 如果复杂度直接消失，这个模块可能只是 pass-through。
- 如果复杂度扩散回多个调用者，这个模块在隐藏复杂度。

这个测试用于判断“该合并、该加深，还是该保留”。

## 依赖分类与测试策略

| 依赖类别 | 例子 | 推荐测试策略 |
|---|---|---|
| In-process | 纯计算、内存状态 | 合并浅模块，通过新接口直接测 |
| Local-substitutable | 本地数据库、文件系统、有本地替身的服务 | 使用本地替身跑集成式测试 |
| Remote but owned | 自己控制的内部远程服务 | 定义接口，生产适配器走网络，测试适配器走内存 |
| True external | Stripe、Twilio 等第三方 | 在外部边界 mock，内部逻辑仍通过模块接口测 |

## 重构方向

1. 找浅模块：接口几乎和实现一样复杂，调用者要知道太多。
2. 做 deletion test：确认它是否真的隐藏复杂度。
3. 收拢规则：把散落在调用者里的规则移入更深模块。
4. 选择接口位置：只在真实变化点建立替换点。
5. 替换测试面：旧浅模块单测可删除或收敛到深模块接口测试。

## 风险与坑

- [确认] 只有一个适配器时，很多接口位置只是间接层，不是结构收益。
- [确认] 为了测试暴露内部结构，会降低接口深度，让调用者知道更多。
- [推断] 本仓库中文文风不适合直接引入英文隐喻；统一使用“接口位置 / 替换点 / 适配器”更贴近现有规则。
