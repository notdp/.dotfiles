# 方案脆弱点类型清单（Fragility Types）

> SSOT。被 `/think-plan`（Premise Collapse）、`/think-research`（Plan Handoff）、`planning-task.md` capsule 引用。
> 解决的反模式：agent 在需求/设计阶段**用猜测代替事实**——识别出方案里有不确定环节，却不先做最小成本的本地验证，直接把未验证假设放进系统级试错，反复改、反复撞。

## 怎么用

需求/方案设计阶段，对照下面 11 类逐条扫描本方案：**命中任一类 = 这个环节是脆弱点，必须先挂一个 spike 把不确定性变成事实，再进入系统级实现。** 不要乐观假设它能成。

这份清单只是识别器；识别后的纪律（挂 spike、spike 通过前不进实现）在 `/think-plan` Premise Collapse 落地。

## Spike 的统一定义（全仓 canonical）

**Spike** = 一次 timeboxed、focused 的调查，唯一目的是**快速降低某个高风险技术不确定性**。

- 动手前**写下这个 spike 要回答的唯一问题**（如「飞书满表写入到底返回什么、能不能扩容」）。
- 定时间盒（如 30 分钟 / 1 小时），**答完即停，不顺手做实现**。
- spike 代码是**丢弃式**的，不要求生产质量、不进 main。
- 产物是**一个事实**（真实错误码 / 真实 response 形状 / 真实配额数字），不是「我觉得应该可以」。

与相邻概念区分（避免术语漂移）：

| 词 | 含义 | 何时用 |
|---|---|---|
| **derisk spike**（本文档） | 丢弃式探索「这个不确定环节能不能成」 | 设计阶段命中脆弱点 |
| **Walking Skeleton / Tracer Bullet** | 保留式骨架，打通端到端最薄真实链路证明「集成点真的通」 | 架构/集成对齐未知 |
| **POC**（think-research） | 证明「某个候选方案可落地」用于选型 | 方案选型阶段，是 derisk spike 的一种 |
| **throwaway spike**（dev-tdd） | 隔离实验理解代码行为，不进 main | 实现期探索 |
| **probe / 观测**（dev-debug） | 加最小观测点让运行时内部状态可见 | debug 阶段（同根，但属运行时而非设计时） |

## 11 类常见脆弱点

| # | 类型 | 为何 agent 易猜错 | 典型例子 | 最小验证（spike） |
|---|---|---|---|---|
| 1 | **SaaS 配额 / 边界 / 满载** | 第三方 SaaS 的硬限制写在文档角落或没写，按套餐/灰度变化；agent 默认「容器无限大」，对「表满/配额耗尽」无心智模型，本地空表永远复现不出 | 飞书 Bitable 单表记录上限、单次批量 ≤500 条；Notion block children 上限；Airtable base 上限；Stripe/短信 rate limit 与日配额；DB 单表/单行字节上限 | 脚本先 GET 当前用量/上限，再在 sandbox 造一个「接近满/已满」的表真打一次写入，记真实错误码；直接读官方 limits 页抄硬数字 |
| 2 | **第三方 API / SDK 实际契约** | 文档与实现常不一致：字段可选性、分页游标语义、错误码、默认值、SDK 隐藏重试超时；agent 按文档「应该如此」写，真实响应形状不同 | 分页有的用 page_token 有的 offset；HTTP 200 但 body code≠0 才算成功（飞书风格）；SDK 默认超时/重试未知叠加上层重试放大；时间/金额单位（ms vs s、分 vs 元） | 最小脚本对真实/sandbox 端点打 1-2 次，dump 原始 response 逐字段比对文档；用官方 reference/cookbook/Postman 集合做基线。**绝不用 mock 验契约**——mock 是你猜的形状 |
| 3 | **鉴权 / token / webhook 回调** | 鉴权是多步握手，任一环节假设错整条链断；回调依赖外网可达、验签、重放/乱序/重试，本地收不到真实回调 | token TTL 缓存过期未刷新→间歇 401；权限 scope 不足返回业务错误码被误判为 bug；webhook 验签算法/challenge 握手各异；回调 at-least-once 重复/乱序投递；回调要求公网可达 | 脚本独立跑完整握手打印每步与 token TTL/scope，故意让 token 过期看真实错误码；回调用 webhook.site/ngrok/控制台「测试推送」抓原始 payload+header 按真实签名验一遍，连发两次验幂等 |
| 4 | **未知技术 / 新库 / 新平台能力** | 对没用过的库/平台只有「文档读起来像能做」的乐观估计，不知在你的具体约束下能否做到、坑在哪；把整套方案押在未验证能力上，集成到一半发现做不到，返工最大 | 「用某 OCR/向量库就能识别/检索」实际精度/语言/格式不达标；「图表库支持这种交互」但你的数据量下卡死；新 SDK 在目标平台（特定运行时/ARM/serverless 冷启动）不可用 | 写 30 行 hello-world POC，只验「这个能力在我的约束下能不能做到、做到什么程度」，不接业务；timeboxed，答完即停。**找不到可运行示例本身就是高风险信号** |
| 5 | **算法 / 数据结构正确性** | 自实现算法边界条件极多，「看起来对」与「真的对」差距大；主路径对、空/单元素/重复/越界/回绕出错；不靠针对性用例几乎发现不了，表现为偶发数据错乱 | 分页游标合并去重漏边界→跨页丢/重复记录；状态机漏非法转移；时间区间开闭判断错；浮点累加/整数溢出/取模负数；自写树遍历缺环检测→死循环 | 本地单测先跑空/单个/最大/重复/越界/逆序边界用例；能用现成库就别自实现（自实现=新风险）；复杂逻辑用 property-based testing 或与朴素参考实现**对拍（差分测试）** |
| 6 | **并发 / 时序 / 竞态** | 经典「本地单线程跑一万次都对、上线偶发」；agent 按顺序执行心智写代码，对竞态/双写/读改写非原子/乱序/重入无默认警觉；不可复现性极强，「猜+改+看」几乎必败 | check-then-act 非原子（先查没有再插→并发双插）；缓存与 DB 双写不一致；消息/回调乱序或重复；多 worker 抢同一任务无锁；异步回调闭包捕获已变更变量 | 写并发压测小脚本并行打 N 次同一操作在本地/sandbox 复现竞态；日志打时间戳+请求 ID 让时序可见；先答「有没有 check-then-act/双写」一个结构问题，命中就上原子操作/唯一约束/锁，而非靠重试掩盖 |
| 7 | **数据形态边界（空/满/超大/编码/脏）** | agent 用「理想样本」（几条干净英文）在脑子里跑方案，真实数据有空值/超长/emoji/CJK/控制字符/嵌套/超大/缺字段/类型漂移；本地理想样本永远撞不到 | 已满的飞书表加行（数据量边界）；非 ASCII/emoji/CJK 截断、按字节算长度超限、编码乱码；空数组/null 空指针；超大 payload 超请求体上限/OOM；脏数据类型漂移/缺字段/重复主键 | 本地用对抗性样本各跑一遍：空/1 个/超大/emoji+CJK/null/超长/缺字段；先抓一份真实数据样本（而非自造）看真实分布；满载类用 #1 手法在 sandbox 造满表实测 |
| 8 | **环境差异（dev/prod、OS、版本、容器、网络）** | 「我本地能跑」是最经典脆弱假设；路径分隔符/时区/locale/依赖版本/CPU 架构/文件系统大小写/网络可达性/serverless 限制任一不同就失败；agent 默认两边一致，把环境差异归因成代码 bug，往错误方向反复改 | 本地 macOS 大小写不敏感、prod Linux 敏感→import/文件找不到；本地 +8、prod UTC→时间全错；依赖小版本不一致行为变；prod 内网连不通本地能访问的外部服务；serverless /tmp 外只读、冷启动超时 | 在尽量贴近 prod 的环境跑最小 smoke（同镜像/架构/版本锁）；用脚本打印环境快照（OS/arch/tz/locale/版本/关键 env/目标可达性）两边各跑一次 diff。**归因环境前必须用工具验证** |
| 9 | **隐含契约（上下游 / shared-path / schema）** | 方案常隐含「调用方会这么传/上游返回长这样/这个字段一定有/改这里不影响别处」；agent 改一处时看不见所有 caller 和下游消费者，破坏隐式约定；集成或回归时才爆，定位跨模块 | 改共享函数返回结构→另一 caller 静默崩；API envelope/schema 变更影响前端反序列化；假设上游永远非空/有序/去重；枚举新增值下游 switch 无 default；config/flag 默认值被另一处依赖 | 先 grep 全部 caller 和字段消费点，把隐式契约写成**显式 contract cases（accept/reject）**再动手（对接仓库 boundary-gate 事实块）；对外部 schema 写最小校验脚本断言关键字段存在性/类型 |
| 10 | **幂等 / 重试 / 副作用安全** | 网络/SaaS 必然有超时和重复投递；agent 默认「调一次成功一次」，没把「重试会不会重复扣款/重复建行」当必答题；加了重试或上游 at-least-once，非幂等操作放大成数据污染 | 「加一行」重试两次变两行；支付/扣费/发消息无幂等键超时重试重复；MQ/webhook at-least-once 消费无去重；批处理中断重跑无断点→部分数据翻倍；重试未区分 5xx/4xx 对 4xx 死循环 | 脚本对同一操作连发两次断言只生效一次（验幂等键/唯一约束/upsert）；模拟超时后重试（打一半 kill）看是否重复；先答「这个写操作幂等吗、幂等键是什么」，没有先加唯一约束/幂等键 |
| 11 | **性能 / 容量 / 规模** | agent 用小数据估算，默认 O(n) 够用、单次拉全量没问题、内存装得下；真实规模下 N+1/全量加载/无索引/同步阻塞/无背压让方案跨过阈值后塌方；小数据指标全绿误导「已验证」 | 循环里逐条调 API（N+1）100 条没事 10 万条打爆 rate limit；一次性 load 全表进内存→OOM；查询无索引行数过百万从毫秒到秒；同步处理大批量阻塞超时；正则在超长输入指数级（ReDoS） | 用接近真实量级（或放大 10-100x 合成）数据在本地跑一次，测时延/内存/调用次数；先做粗量级估算（N×单次成本）看是否跨过 rate limit/内存/超时阈值 |

## Spike 通用手法

- **最小脚本打真实/sandbox API**：20-50 行打 1-2 次，dump 原始 response 与 header，用事实推翻你对契约/分页/错误码的猜测——**绝不用 mock 验契约**
- **造对抗性数据复现边界**：本地构造 空/单个/满载/超大/emoji+CJK/null/缺字段/重复主键 各跑一遍；满载类用 SaaS sandbox 造满表实测
- **直接读官方 limits/quotas/error-code 页**：抄硬数字（单表上限、单批条数、rate limit、TTL、错误码语义），而非二手博客或凭记忆
- **timeboxed spike**：先写下「这个 spike 要回答的唯一问题」，定时间盒，答完即停
- **对拍 / 差分测试**：自写算法时与朴素实现或现成库对同一批输入比对结果，自动暴露边界分歧；配合 property-based testing
- **并发压测小脚本**：并行打 N 次同一操作复现竞态，而不是等线上偶发
- **幂等探针**：对同一写操作连发两次 + 模拟超时后重试（打一半 kill），断言只生效一次
- **环境快照对比**：脚本打印 OS/arch/tz/locale/依赖版本/关键 env/目标可达性，dev 与贴近 prod 环境各跑一次逐项 diff
- **贴近 prod 的 smoke**：同镜像/架构/版本锁跑端到端最小路径，证明集成点真的通（walking skeleton）
- **鉴权/回调隔离跑通**：独立脚本跑完整鉴权握手打印每步与 TTL/scope；回调用 webhook.site/ngrok/控制台测试推送抓真实 payload 按真实签名验一次
- **规模放大测量**：用接近真实量级（或放大 10-100x）数据测时延/内存/调用次数，再做 N×单次成本量级估算
- **把隐式契约写成显式 contract cases**：grep 全部 caller 与字段消费点，列 accept/reject 用例再动手

## 方法论锚点

- **Spike（Agile/XP）**：timeboxed、focused 调查，唯一目的快速降低不确定性；丢弃式，答完即弃。命中脆弱点即先挂 spike 再实施。
- **Tracer Bullet（《务实程序员》）**：从入口到产出打通最薄端到端真实链路（steel thread），生产级但精简，验证各架构组件能否对齐。保留式骨架，后续在其上长肉。
- **Walking Skeleton（Cockburn）**：实现能端到端跑通的极小系统，串起主要组件，先证明「集成点真的通」，把「与外部世界协作」的风险前置（front-load risk）。
- **De-risking / Risk burn-down**：项目风险应随时间下降而非留到最后；识别最高不确定环节（外部依赖/未知技术/并发/规模/隐含契约）优先攻克，把「未知未知」尽量转成「已知」。这是对抗 planning fallacy（计划性乐观偏差）的核心手段。
- **统一根因**：脆弱点的本质是「用猜测代替事实」。方法论统一落点——**用最小成本把不确定性变成可见事实（spike 打真实 API / 加观测探针），再决策**；这同时覆盖设计阶段的「猜方案」和 debug 阶段的「猜根因」。

## 与 CLAUDE.md risk types 的对应

本清单是「实施前需先验证的技术不确定性」视角，与全局 `Boundary facts` 的 risk types（边界决策需声明的视角）部分重叠但侧重不同。做边界决策时仍用 risk types 词表；这里是其技术细化，不另起一套并列分类：

| 本文档脆弱点 | 对应/相关的 risk type |
|---|---|
| #1 SaaS 配额 / 满载 | `limit-default-fallback` |
| #2 第三方 API 契约、#9 隐含契约 | `schema-contract` / `shared-path` |
| #7 数据形态边界 | `data-source` / `limit-default-fallback` |
| #10 幂等 / 副作用 | `operational-side-effect` |
| #1/#3 涉及生产/凭据/外部写 | `operational-side-effect`（并按需走 `/guard-gitops`） |

#4 未知技术、#5 算法正确性、#6 并发、#8 环境、#11 性能 主要是**技术可行性不确定性**，risk types 未直接覆盖——正是本清单的增量。
