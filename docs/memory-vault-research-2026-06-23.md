## 综合设计骨架

### 集成总览

设计为「与现有 context-capsule 同构的第二条 capsule 流 + 复刻 dev-long-run 工作区的统一 vault」，零新基础设施、纯文件+复用现有 hook/plugin 管道。三层定位（对齐 CoALA / Letta）：(A) 规则层 procedural = agents/AGENTS.md、hooks，仅人改，agent 永不自动改写（CoALA 明确 procedural 自写是 bug/越权风险，confirmed: arxiv 2309.02427 L689-692）；(B) memory 层 semantic/episodic = agent 可写的跨 session 知识，落 git-tracked 文件；(C) vault 层 = per-task 本地过程态，gitignored。

数据流：
1. 读注入（每 prompt 开局）：复用 context_capsule.py 的 json_context() 协议（cc/droid/codex 走 additionalContext + systemMessage 可观测一行）与 dotfiles_hooks.mjs 的 chat.message 文本追加（kilo/opencode，新增 MEMORY_MARKER 与 CAPSULE_MARKER 并存），grep-first 检索 top-N memory 索引摘要注入，独立预算不挤占现有 2200 字符（confirmed: context_capsule.py:84 MAX_PROMPT_CONTEXT_CHARS=2200, :235 json_context, dotfiles_hooks.mjs:49-64）。
2. 写捕获（Stop / session.idle，5 平台最低公分母）：Stop hook 复用 context_state.py 骨架（transcript 抽取 + redact() + TTL + 分桶落盘）做纯本地「候选捕获」到 staging，零价值判断、fail-open、不调 LLM（confirmed: context_state.py:90 redact, :20 TTL, install_hooks Stop 5 平台齐）。
3. 提纯固化（skill）：assist-learn 式 skill 由模型把候选 staging 提纯为正式 memory（ADD/UPDATE/DELETE/NOOP 去重，借 mem0 写纪律对抗 append rot），落 docs/learnings 既有 store + 新增 user-level store。
4. vault：dev-long-run/dev-complete 的 .long-loop/<date>_<slug>/ 即统一 vault 实例，内部分 machine/(ephemeral) 与 reviewable/(人审持久)，agentsview 文件级消费 machine 层 JSONL/JSON。

不引入：向量库、embedding、常驻进程、MCP server（Chachamaru harness_mem_*/SuperClaude Serena 都是 CC/Codex 中心、违反 agent-agnostic + 避免重基础设施）。grep 检索在数千文件 / 1MB jsonl 内 sub-10ms，benchmark 证据(muratcankoylan: 文件系统 74% LoCoMo > Mem0 68.5%)支撑「无向量库」。

### 存储设计

三个存储根，按生命周期与可见性分离（复刻已验证约定，不发明新结构）：

1. 用户级 memory（跨项目、需跨机器同步）→ ~/.dotfiles/memory/user/ 下纯 markdown（git-tracked，满足 SSOT 纪律「仓库内才算存在」+ 解决 auto-memory machine-local 换机即丢的隐患），经软链 fanout 到 5 agent config 目录（复用 install_hooks agent_asset_links 模式，confirmed: install_hooks.py:653-674 已把 AGENTS.md/skills 软链到 .codex/.factory/opencode/kilo/~/.claude）。结构 = MEMORY.md 索引（一行摘要+相对链接，CC auto-memory 已验证的索引模式）+ 单文件单事实 topic notes。

2. 项目级 memory（跨 session、可 commit、跨 agent 可见）→ 沿用 assist-learn 既有 canonical store <repo>/docs/learnings/<category>/<slug>.md（不另起炉灶，confirmed: assist-learn/SKILL.md:23）。这是已存在的 agent-agnostic project memory，新设计收编它而非造第三套。

3. per-project vault（per-task 本地过程态）→ <repo>/.long-loop/<YYYYMMDD>_<slug>/（gitignored，复刻 dev-long-run 事实标准，confirmed: lr.py workspace + .gitignore .long-loop/）。

frontmatter 固定一套 schema（避免 CC 嵌套 metadata vs 平铺 type 的漂移）：title/date/type(枚举: user|feedback|project|reference，4 值已实践验证)/module/component/tags/origin_session/review_condition。高频短条目（error→fix reflexion、metrics）可选用 append-only JSONL 带 {_schema,_version} 头行（SuperClaude/digital-brain 模式，对 hook 写入无 read-modify-write 竞态、agentsview 易消费）；curated 知识用单文件 markdown。

格式选择决策：两层——hook 写 JSONL staging（race-free、可追加），skill 提纯为 markdown 知识。

### 检索/读取

grep-first 关键词加权，无 embedding（用户已明确反对分类器 gating 漏召回，且 grep 可解释/fail-open/零基础设施）：

- 注入侧（hook，每 prompt）：复用 learnings_search.py 的打分（frontmatter 命中 ×3、body ×1，confirmed: FRONTMATTER_WEIGHT=3），对当前 prompt 关键词检索，只注入 top-N 的 path + frontmatter 摘要（不 dump 全文），经 context_capsule 同款注入通道。读取时附 mtime 派生 age 标签 + 「memory 是补充，code+对话优先，与现有代码冲突时以代码为准」核验提醒（复刻 CC age system-reminder + EveryInc Phase0.5 read-time contract）。
- 深查侧（skill opt-in）：debug/plan/research 前模型主动跑 learnings_search 全文检索读 Reusable Pattern（assist-learn 已有此约定）。

检索模型借 Generative Agents 3 因子但全部降级为「写时编码 + grep 排序」：recency=文件按日期排序读 tail；importance=写入时 LLM 写的 frontmatter tag；relevance=grep 关键词匹配。不引入运行时 scorer/decay 进程。

「宁多召回让模型自筛」对抗关键词字面匹配弱点：靠索引文件 + 结构化 heading + 显式 tag 把负担前移到写入纪律。语义召回缺口（约 7% NDCG，inferred from RAG 文献）是刻意取舍。

EveryInc Discoverability 纪律：在 agents/AGENTS.md 用描述性（非命令式「always search」会致冗余读）一句提示 5 agent 知道有此 store。

### 写入

两段式混合（hook 廉价捕获 + skill 提纯固化），不是 hook/skill 二选一——单用各有致命缺口（hook 无价值判断易记流水账；skill 不保证触发，~50-80% 命中率，confirmed: affaan continuous-learning「Why Hooks vs Skills」），缺口互补：

1. hook 候选捕获（Stop / session.idle，5 平台齐，唯一可靠+跨 agent 写路径）：复用 context_state.py 的 transcript 正则抽取 + redact() 脱敏 + TTL + 分桶，落 staging JSONL（candidates/），确定性、fail-open、纯本地不调 LLM（hook 硬约束：快/不阻塞，confirmed: install_hooks timeout 10/15）。保证「不漏」。droid 仅 Stop（无 PreCompact/SessionEnd），所以必须 Stop 即落盘不押后到 compact（confirmed: harness-ops 矩阵 droid PreCompact ❌）。

2. skill 提纯固化（assist-learn 式，5 平台都能调 markdown+脚本）：模型判断哪些候选值得固化，按 mem0 ADD/UPDATE/DELETE/NOOP 对已有条目去重/解冲突（对抗 append rot，confirmed: arxiv 2504.19413 L157-160），提炼「真正学到的模式」写 docs/learnings/ 或 user store。保证「质量」。

3. 用户显式 /assist-learn 是最高优先写入路径。

谁决定记什么：三档明确——hook 机械捕候选（不判价值）/ skill 模型判价值固化 / 用户显式。CoALA 红线：memory 写入永不触碰 procedural 规则层（agents/*.md、hooks），规则改动人工 gated。写入前强制 redact()（注意现有正则仅覆盖 sk-/api_key/token/password/secret，覆盖不全是已知风险）。

compact-时捕获（codex PreCompact / opencode experimental.session.compacting）作为 cc/codex/oc/kilo 增强，droid 显式降级登记「仅 Stop 捕获」（对齐 cc-native-adoption-policy 差异化记录不假设平价）。

### hook vs skill

两者都用，按可靠性 vs 价值判断分工，这是 findings 一致结论（affaan 实证 + 本仓库现有分工）：

- hook 负责确定性触发/廉价捕获：写侧 Stop 抓候选（hook 100% fire vs skill 50-80%）；读侧 UserPromptSubmit 注入索引（被动可靠、跨平台、模型不会忘记看历史）。hook 必须快/fail-open/advisory，所以只做正则抽取 + 本地文件 IO + grep，绝不在 hook 内调 LLM 做摘要/提纯（违反快约束，inferred 但有 timeout 证据支撑）。
- skill 负责价值判断/高质量：写侧把候选提纯为正式 memory（LLM 推理 + ADD/UPDATE/DELETE/NOOP）；读侧 opt-in 深查全文。

核心论证：纯 hook 记不出「真正的模式」（context_state.py current_step 写死 'not inferred' 证明 hook 无推理，confirmed）；纯 skill 多数会话静默不触发→memory 稀疏复利断。hook 给广度兜底、skill 给深度与质量。

读取「可靠」= 在 always-loaded 规则文件里写一句「开局读 memory 索引」的指令（5 平台通用，最可靠的读）+ hook 注入（4/5 平台 prompt-aware；codex 退化为静态 AGENTS.md 注入并显式记录）。写入「可靠」= hook 触发 + skill 提纯，不能只靠 read-only 指令（agent 在 compact 下会忘记写回）。

### 用户级 vs 项目级

用 affaan 的 git-remote-hash 项目 ID 做 user-vs-project key（同 repo→同 ID 跨机器稳定，fallback repo path→global，confirmed: continuous-learning-v2 L126-146）。两级是两个文件 scope，不是同一目录的标签：

- 用户级 memory（type: user/feedback 等跨项目可复用）→ ~/.dotfiles/memory/user/，git-tracked + 软链 fanout 5 agent。注意 CC 原生 auto-memory 根本没有真用户级（全 per-repo，dotfiles 里 type:user 条目其实困在 dotfiles repo 目录、对 ordo-ai 不可见，confirmed）——真用户级持久层历来是 ~/.claude/CLAUDE.md=AGENTS.md。新机制把用户级 memory 显式落 ~/.dotfiles 固定目录补这个洞。
- 项目级 memory（type: project/reference，含本项目决策/调研事实）→ <repo>/docs/learnings/，随项目 repo commit。
- vault → <repo>/.long-loop/（本地）。

晋升规则（项目→用户）：借 affaan 自动晋升思路（同一 instinct 在 2+ 项目出现且置信≥0.8）但降级为人工/skill 触发（无后台进程），由 /assist-learn 或专门 skill 把项目级反复出现的事实搬到 user store。scope 决策表（confirmed: continuous-learning-v2 L242-303）：language/framework/file-structure/code-style→项目；security/git/tool-workflow/general-best-practice→用户。

读时冲突（user vs project 矛盾）：默认 project 胜（更具体、更近上下文），注入时同时呈现并标注，由模型在 prompt 中裁决，不做文件层硬合并。[推断]

### memory 与 vault 边界

边界清晰且 findings 一致：

- vault = per-task 工作过程态，短生命周期（任务完即归档/可丢弃），gitignored 纯本地（不跨机器/不进 PR），落 <repo>/.long-loop/。是「我这次怎么做的」的过程产物：spec/plan/status/verify.json/handoff/logs/metrics（confirmed: lr.py 工作区）。
- memory = 跨 session/跨项目长期知识，长生命周期，git-tracked（要跨机器/跨 agent 同步、可审计），落 docs/learnings（项目级）或 ~/.dotfiles/memory/user（用户级）。是「我学到了什么/我的偏好/已确立的决策」。

关键差异（必须显式划清，否则冲突）：vault 照搬 .long-loop 的 gitignore=本地不可同步；memory 反之需 tracked/可同步——所以 memory 不能照搬 vault 的 gitignore 策略（confirmed risk: 调研 5/6 都点出此边界）。

晋升机制（vault→memory）：vault 内沉淀的 decision+why、reusable pattern 在任务结束时，由 skill（/assist-learn 或 vault 归档钩子提示）判断哪些值得「晋升」为正式 memory，搬到 docs/learnings。这是防止 vault 无限堆积 + memory 被任务噪声污染的关键闸门。晋升应绑可验证 artifact（commit/verify.json），避免重蹈「声明完成但没验证」（confirmed: lr 完成门禁背景 00167）。

rule-vs-state 分离（贯穿两者）：规则文件（AGENTS.md/ASSERT.md）只放规则+命令，状态进 progress/logs/memory，防 always-loaded 层膨胀漂移（confirmed: long-running-agent L221）。

### vault 设计

直接复刻 dev-long-run 已验证的 .long-loop/ 工作区为统一 vault（事实标准，dc.py 已 import lr.py 共用同根，confirmed: dc.py:127, lr.py:958）。统一 vault = <repo>/.long-loop/<YYYYMMDD>_<slug>/，纯文件约定非新基础设施（满足避免重基础设施约束）。

内部显式分两轨（沿用现有 machine/人审双轨持久化，confirmed: lr.py:230-322）：
- machine/（ephemeral，可重建、agentsview 消费面）：state.json（state/phase/branch 进度指针）、*.status（单行 done commit=<hash>/coding/blocked）、verify.json/acceptance.json（exit+output_tail）、stuck.json（失败指纹计数，跨 compact 不丢）、metrics.jsonl（append-only 事件流，纯增量零回改）、SESSIONS.md（pane 注册表）。
- reviewable/（人审持久，可晋升为 memory）：SPEC_OVERVIEW.md+Coverage Matrix、fix_plan.md（phase checklist）、HANDOFF.md（fresh coder 续接）、review_*.md、ack.md（含 [fixed]/[rejected] 裁决+理由）、qa.md（人工验证三段：目的/操作/观察）、BACKLOG.md、CLEANUP_PROPOSAL.md、decisions.md（decision+why，新增，绑可验证 commit/verify.json，loop closed 经门禁标注非口头）、logs.md（append-only）。

容纳两形状：dev-complete 单层扁平（无 phases/）+ dev-long-run 多 phase（phases/<NN>_<slug>/ 作为可选子层），根目录放跨 phase 滚动态。统一 schema 用「可选 phases/ 子层」兼容两者，不强行单一结构割裂其一（confirmed risk）。

断点续跑三件套（state.json + checklist + dead-session reconcile，记忆不可信，confirmed: lr.py:1183-1213）+ 完成门禁（complete-phase/complete-run gate-only state transition，commit 真在分支 git merge-base --is-ancestor，confirmed: lr.py:325-391）作为强制 Judge/Review 元素（codex-maxxing: 无 Review 则 Memory 只记录 agent 自称完成）。映射现有 guard-verify/guard-check 为 Judge。

vault decision+why 与 loop-closed 标注尽量绑可验证 artifact（避免重蹈 00167）。

显式不可用登记：codex 在旧矩阵下注入退化为静态 AGENTS.md（若官方 hook 实测可用则升级）；droid 写捕获仅 Stop。

## 待用户拍板的决策点

### D1. 用户级 memory 的物理落点与是否进 git？

- ~/.dotfiles/memory/user/ 内(git-tracked，满足 SSOT『仓库内才算存在』+ 跨机器同步 + 可被 agentsview git 消费，但 memory 含工作内容会进版本库)
- ~/.config/ 共享目录(XDG，不进 git，隐私好但违反 SSOT、不跨机器、agentsview 难发现)
- 混合：索引+非敏感事实进 ~/.dotfiles，敏感/工作内容留本地 ~/.config

**推荐**:~/.dotfiles/memory/user/(git-tracked)。理由：解决 auto-memory machine-local 换机即丢的现存隐患、满足 SSOT 纪律、复用现成软链 fanout、天然可观测。隐私靠写入前强制 redact() + 写入纪律(不记 secret/客户数据)兜底，而非靠不进 git。

### D2. 是否收编/迁移已有 50+ 条 CC auto-memory(dotfiles 8 + ordo-ai 24 等)进新机制？

- 全量迁移(一次性脚本把 ~/.claude/projects/*/memory 转成新 schema 落 repo)
- 只管未来，旧的留给 CC auto-memory(新旧并存)
- 按价值人工挑选迁移高价值条目

**推荐**:按价值人工挑选迁移(优先 dotfiles 这 8 条 harness 知识到 ~/.dotfiles/memory/user/)。全量迁移有 frontmatter 嵌套/平铺漂移解析成本且会搬入噪声；纯不迁会让其它 4 agent 永远看不到这些用户级知识。先迁高价值小批量验证机制，再决定批量。

### D3. memory 与 vault 是否共用 .long-loop 存储根？

- 合并：memory 也放 .long-loop(目录统一但 gitignore 策略冲突——memory 需 tracked)
- 分离：vault 在 .long-loop(本地)，memory 在 docs/learnings + ~/.dotfiles(tracked)
- vault 内设 memory-candidates 暂存，晋升后搬到独立 memory store

**推荐**:分离 + vault 内 candidates 暂存(选项3 叠 2)。vault=gitignored 本地过程态、memory=tracked 长期知识，gitignore 策略根本冲突不能合并根；但 vault 可设 candidates/ 作为 hook 候选捕获的暂存，由 skill 晋升搬到 memory store。

### D4. hook 候选捕获用 JSONL staging 还是直接写 markdown？

- JSONL staging(race-free 追加、agentsview 易消费，但需 skill 二段提纯)
- 直接 append markdown(少一段，但 hook 无价值判断会污染检索 + read-modify-write 竞态)
- 只标记不写内容(hook 仅记『此 session 有候选』，全靠 skill 回捞 transcript)

**推荐**:JSONL staging。append-only 无竞态、自带 _schema 头可观测、把价值判断留给 skill。直接写 markdown 会让低价值流水账进检索面；只标记则丢失 hook 廉价捕获的确定性价值。

### D5. consolidation/防膨胀触发方式(无 daemon)？

- 快 hook 检测文件行数/大小超阈值时 nudge(参照 CC 200 行/25KB)
- 纯用户/skill 显式调用(/compact-memory)
- 两者：hook 软提醒 + skill 执行

**推荐**:两者(hook 软提醒 + skill 执行)。hook 只检测大小贴一行 advisory 提醒(快/不阻塞)，实际 LLM 压缩(episodic→semantic)由 skill 做——绝不在 hook 内调 LLM(违反快约束)。对齐已有 fragility-spike 的 soft/advisory 纪律。

## 矛盾点

- harness-ops 平台矩阵 vs codex 官方文档：旧矩阵记载『codex 无 hook 体系 / PreCompact ❌』，但 codex 官方文档(WebFetch 2026-06, inferred)显示已支持完整生命周期 hooks(含 SessionStart/PreCompact/PostCompact + additionalContext)与原生 memories。基于旧矩阵会给 codex 设不必要降级路径。须实测复核后更正矩阵再设计。
- one-file-per-learning markdown(EveryInc/assist-learn，适合 curated 检索) vs append-only JSONL(SuperClaude，适合高频 reflexion + race-free hook 写 + agentsview 消费)——两路调研各推一种。综合判定为两层(hook 写 JSONL staging + skill 提纯为 markdown)，但『晋升门』阈值未定。
- 写入触发『谁决定记什么』：hook 自动抽取保证不漏但质量天花板低(context_state.py current_step 写死 not inferred 证明 hook 无推理)；skill 模型判断质量高但只 50-80% 触发。两者单用都被各自调研判为有致命缺口——非真矛盾，综合为两段式互补，但具体触发频率/阈值需 spike。
- 检索匹配：deepseek 分类器路由(capsule 现用，F1 54%) vs grep 关键词打分(learnings_search)。用户已明确反对分类器 gating(漏召回)，故 memory 检索定为 grep-first；但分类器作为可选增强 vs 完全弃用，调研间倾向不一(D 推 grep-first，部分推可复用 deepseek 范式)。综合取 grep-first，分类器最多可选不做 gate。

## 风险

- append-only memory rot：若 hook 直接 append 且无 mem0 式 ADD/UPDATE/DELETE/NOOP 去重 + 周期 consolidation，store 会堆积陈旧/矛盾条目，拖慢检索并误导 agent(confirmed gotcha: muratcankoylan stale-memory-poisoning + ce-learnings-researcher『never let past learning silently override present evidence』)。缓解=两段式 skill 提纯去重 + read-time『code 优先』契约 + consolidation pass。
- procedural 自改写危险：若 auto-memory 越界改规则层(agents/*.md、hooks)而非沙箱化的 memory/vault，harness 会静默漂移自身护栏(confirmed: CoALA L689-692)。硬约束=规则层人工 gated，memory 永不写规则。
- 脱敏遗漏致 secret 进 git：自动抽取写入若漏 redact 会把 secret/URL 落进 tracked memory；现有正则仅覆盖 sk-/api_key/token/password/secret，覆盖不全(confirmed: context_state.py:21 SECRET_RE)。须强制走 redact() + 扩充正则 + git-policy(决策/pattern 共享、observations/session 本地)。
- 写入触发覆盖不齐：hook 在 5 agent hook 面不同(cc/droid 原生 vs kilo/opencode .mjs vs codex)，capture 覆盖会不均；droid 仅 Stop 无 compact 钩子，长会话被 compact 时无法拦截落盘(confirmed: harness-ops 矩阵)。缓解=Stop 即落盘不押后、capture 也做成用户可调 fallback、差异化登记。
- kilo/opencode 注入崩 session：chat.message 若 push 新 part(缺 id/messageID/sessionID)会让 session 崩 UnknownError 且 try/catch 抓不到(confirmed: 记忆 kilo-opencode-plugin-hooks 实测)。memory 注入必须追加既有 text part 并 opencode run 端到端验证(spike 自造 parts 假通过)。
- 注入预算挤占 + 噪声：现有 capsule 已占 2200 字符；叠加 memory 注入会抢预算且每 prompt 注入相关 memory 可能分散注意力(用户已对常驻层堆内容警惕，governance A4 VOI 被搁置)。须独立预算 + 只注入索引摘要(非全文) + top-N 限制。
- discoverability 双向风险：store 无 agent 读则复利为零(EveryInc 专设一阶段)；但命令式『always search』致冗余读。须在 AGENTS.md 描述性(非命令式)一句提示，校准。
- vault 形状分裂：dev-long-run 多层 phases/ vs dev-complete 扁平，强行单一 schema 会割裂其一(confirmed risk)。缓解=phases/ 作可选子层。
- agentsview 采集路径/schema 未知(external)：vault 多为 gitignored .long-loop，agentsview 怎么发现并读到未核；input schema 在另一 repo 未读。固定 vault emitted schema 前须对齐，否则预留的消费面可能不匹配。
- 机器态噪声爆炸：单 vault 已数十文件含截图，若全量喂 agentsview 或全升人审会噪声爆炸。须界定 machine/(可丢) vs reviewable/(持久)，retention 用 14 天 TTL 先例。

**建议**:推荐【两段式混合】，不是 hook 或 skill 二选一——它们各自单用都有致命缺口(hook 无价值判断/skill 不保证触发)，但缺口互补。\n\n【写入：hook 廉价捕获 + skill 提纯固化】(1) hook 侧(Stop 事件，5 平台齐)做'候选捕获'：复用 context_state.py 的 transcript 抽取+redact()+落盘到一个 candidates/ 暂存区(JSON)，确定性、fail-open、不做价值判断。这保证'不漏'。(2) skill 侧(assist-learn 已是现成范式)做'提纯'：模型判断哪些候选值得固化为正式 memory(docs/learnings/<cat>/<slug>.md，带检索 frontmatter)，提炼'真正学到的模式'而非流水账。这保证'质量'。(3) 用户显式 /assist-learn 仍是最高优先写入路径。理由：hook 解决'该记的时刻模型可能没意识到'(触发可靠性)，skill 解决'hook 抽不出价值'(质量)。\n\n【读取：hook 自动注入 + skill opt-in 深查】(1) hook 侧(UserPromptSubmit，5 平台齐，直接扩展 context_capsule.py 的注入机制)：用 learnings_search 的 grep keyword 打分(非分类器，避免用户已反对的 gating 漏召回)对当前 prompt 检索 top-N memory，只注入 frontmatter 摘要+path(不 dump 全文)，受独立预算约束(不挤占现有 capsule 的 2200 字符)。这做到'prompt 开始就有相关历史'。(2) skill 侧 opt-in 深查(learnings_search.py 已有)：模型在 debug/plan/research 前主动跑全文检索读 Reusable Pattern。理由：hook 给广度兜底(模型不会忘记看历史)，skill 给深度(模型判断需要细读哪条)。\n\n【为什么必须混合】纯 hook：确定性触发但 LLM 推理在 hook 内受限(只能正则/分类)，记不出'真正的模式'、检索召回靠关键词易偏。纯 skill：能推理但依赖被调用，多数会话静默不触发→memory 稀疏、复利断。混合让确定性触发(hook)和价值判断(skill)各司其职。\n\n【强约束】① 不依赖 CC 原生 memory(CC-only，违反采纳政策)，用 docs/ markdown + 脚本自建跨平台层。② 写入触发用 Stop 不用 PreCompact(droid/codex 无 PreCompact)。③ oc/kilo 注入只能改既有 text part 不能 push 新 part(否则崩 session)。④ 不引入向量库/embedding/常驻进程(grep 打分已够，符合 markdown-prompt harness)。⑤ 写入前强制 redact()。⑥ 把 assist-learn 现有 store 当作 project-level memory 的 SSOT，不另起炉灶。\n\n所有结论基于亲自读到的源文件(confirmed)；'两段式混合'的具体阈值/触发频率为[推断]，需 spike 验证 hook 内 grep 检索延迟和注入噪音(参照 governance 的'先廉价验证再实施'纪律，fragility-spike-observability-discipline)。

### External research: mature agent long-term memory architectures, downgrade-mapped to a markdown-prompt + filesystem harness (no vector DB / no daemon). All findings are EXTERNAL — sourced from papers/docs/blogs, NOT verified to hold in ~/.dotfiles. Treat as inspiration only.

**关键发现**

- [confirmed] CoALA is the canonical taxonomy splitting agent long-term memory into episodic (past behavior sequences/experiences), semantic (facts about world+self), procedural (the 'production system itself' — rules/code that determine behavior), plus working memory (in-context). Key transferable idea: a language agent can WRITE new knowledge from LLM reasoning over episodic experience INTO semantic memory 'as a form of learning' — legitimizes 'reflect on raw logs -> write distilled rules'.
  - 证据:arxiv.org/pdf/2309.02427 (Sumers/Yao/Narasimhan/Griffiths, TMLR 02/2024), read directly: L265-268 'Procedural memory stores the production system itself: the set of rules... Semantic memory stores facts about the world, while episodic memory stores sequences of the agent's past behaviors'; L682-683 'language agents may also write new knowledge obtained from LLM reasoning into semantic memory as a form of learning'
- [confirmed] CoALA explicitly warns writing to PROCEDURAL memory (agent's own rules/code) is 'significantly riskier than writing to episodic or semantic memory, as it can easily introduce bugs or allow an agent to subvert its designers' intentions', and procedural memory must be designer-initialized. Mapping: agent auto-memory should default to semantic/episodic facts (a MEMORY.md log), NOT auto-rewrites of the rule layer (agents/*.md, hooks); rule-layer edits stay human-gated.
  - 证据:arxiv.org/pdf/2309.02427 read directly: L689-692 'Unlike episodic or semantic memory... procedural memory must be initialized by the designer... while learning new actions by writing to procedural memory is possible, it is significantly riskier... as it can easily introduce bugs or allow an agent to subvert its designers' intentions.'
- [confirmed] Generative Agents (Park et al. 2023) retrieves from a flat natural-language memory stream via 3-factor weighted score: score = a_recency*recency + a_importance*importance + a_relevance*relevance. CONFIRMED specifics: recency = exp decay factor 0.995 over hours since last RETRIEVAL (not creation); importance = LLM outputs integer 1-10; relevance = cosine similarity of embeddings; all weights alpha = 1; each factor min-max normalized to [0,1]; top-ranked memories that fit context are injected.
  - 证据:ar5iv.labs.arxiv.org/html/2304.03442 fetched directly: 'decay factor is 0.995'; importance 'output an integer score' 1-10; relevance = 'cosine similarity between the memory's embedding vector and the query'; 'all alphas are set to 1'; '[0,1] via min-max'
- [confirmed] Generative Agents reflection = periodic consolidation: triggered when SUM of importance scores of recent events exceeds threshold 150 (~2-3x/day); pulls ~100 most recent records, asks LLM for the 3 most salient high-level questions, synthesizes higher-level insights written BACK as new (recursively reflectable) memories. Prior art for 'distillation as scheduled/threshold-triggered batch job', not per-turn.
  - 证据:ar5iv.labs.arxiv.org/html/2304.03442 fetched: 'we generate reflections when the sum of the importance scores for the latest events... exceeds a threshold (150)... reflected roughly two or three times a day'; WebSearch paper summary: '100 most recent memory records... suggest 3 high-level questions... insights added to memory'
- [confirmed] MemGPT/Letta = OS virtual-memory analogy with 3 tiers: Core (always in system prompt; persona+human blocks; agent self-edits via core_memory_replace/memory_rethink/memory_insert tool calls), Recall (full conversation history in DB, tool-searched), Archival (vector store, insert/search via tool). Agent decides paging via TOOL CALLS — memory management is an explicit agent action, not implicit. Transferable: small always-loaded 'core' + larger searchable cold store + agent-invoked read/write.
  - 证据:lin-guanguo.github.io/llm-memory-research/letta.research/ fetched: 'Core Memory (RAM)... always embedded in the system prompt via editable Blocks'; tools core_memory_replace()/memory_rethink()/memory_insert(); 'agents possess write-access to their own prompts'; WebSearch: 'Core (always in-context) -> Recall (searchable history) -> Archival (vector storage)'
- [confirmed] mem0 separates a two-phase write pipeline: (1) EXTRACTION — LLM pulls salient facts from a new message pair + context; (2) UPDATE — LLM compares each candidate fact to semantically-similar existing memories and emits one of four ops via tool/function call: ADD (new), UPDATE (augment), DELETE (contradicted), NOOP (no change). This dedup/conflict-resolution write discipline counters append-only memory rot. Reported: 26% LLM-as-Judge gain over OpenAI Memory, 91% lower p95 latency, >90% token savings vs full-context (LOCOMO).
  - 证据:arxiv.org/pdf/2504.19413 (Chhikara et al.) read directly: L157-160 'four distinct operations: ADD... UPDATE... DELETE... NOOP'; L127-129 extraction on message pair (mt-1,mt); L20-23 '26% relative improvements in LLM-as-a-Judge over OpenAI... 91% lower p95 latency and saves more than 90% token cost'
- [inferred] Retrieval tradeoff (external consensus): BM25/keyword needs no model/GPU/API, is millisecond-fast and fully explainable but only matches literal terms; vector adds 100-300ms + embedding infra and is opaque but catches semantic equivalence; HYBRID (run both, merge/re-rank) is the production standard, ~7% NDCG above either alone. **可复用模式**

- **3-tier memory (core / recall / archival)**(来源:MemGPT/Letta — lin-guanguo.github.io/llm-memory-research/letta.research/):Tiny always-in-context 'core' block (identity + hot facts) + larger searchable warm store (history) + cold long-term store; agent pages between them via explicit read/write tool calls. → 适用:DOWNGRADE: core = a small MEMORY.md head auto-injected each session (cap it, e.g. CC's 200-line/25KB precedent); 'recall/archival' = rest of MEMORY.md + per-project vault files retrieved on demand via grep/glob (no vector DB). 'Self-edit via tool call' downgrades to a skill the agent invokes to append/update, since kilo/codex lack native memory tools.
- **Two-phase write with ADD/UPDATE/DELETE/NOOP**(来源:mem0 — arxiv.org/pdf/2504.19413 L152-160):Extract salient facts, then for each fact decide ADD/UPDATE/DELETE/NOOP against existing similar memories to dedup and resolve contradictions instead of blind append. → 适用:DOWNGRADE: the single most directly portable idea. A 'remember' skill greps existing MEMORY.md entries by topic/key, then instructs the LLM to pick ADD vs UPDATE-in-place vs DELETE-stale vs NOOP — kills append-only rot with zero infra. Conflict detection = the LLM, not a graph DB.
- **3-factor retrieval (recency x importance x relevance)**(来源:Generative Agents — ar5iv.labs.arxiv.org/html/2304.03442):Rank memories by weighted sum of recency (exp decay 0.995 since last access), LLM-rated importance (1-10), embedding relevance; all weights = 1, min-max normalized. → 适用:DOWNGRADE: drop embedding-relevance for keyword/grep relevance (BM25-spirit). Encode recency as file ordering / dated entries (newest last, read the tail). Encode importance as an LLM-authored tag at write time (e.g. [importance: high] or a priority section) — so 'scoring' becomes authoring discipline + grep ordering, not a runtime scorer. The factor MODEL transfers even though the arithmetic doesn't.
- **Reflection / periodic consolidation**(来源:Generative Agents — ar5iv... (threshold 150, ~2-3x/day) + CoALA 'reason over episodic -> write semantic'):On threshold/schedule, batch-read recent raw memories, ask LLM for top salient questions, synthesize higher-level insights, write them back as new (reflectable) memories. → 适用:DOWNGRADE: a /compact-memory or assist-learn-style SKILL run on demand (or nudged when MEMORY.md exceeds N lines), not a daemon. Reads accumulated episodic entries, rewrites into fewer semantic rules — mirrors existing assist-learn/assist-retrospect skills. Threshold-triggered, not per-turn — keeps hooks fast.
- **Stable-rules vs learned-memory separation**(来源:AGENTS.md/CLAUDE.md practice — augmentcode.com + humanlayer.dev + orchestrator.dev):Human-authored hard rules live in a stable file (CLAUDE.md/AGENTS.md); agent-learned preferences/patterns live in a separate writable file; reliably-obeyed rules get converted to hooks. → 适用:DIRECTLY APPLICABLE: maps onto CoALA's procedural(human-gated) vs semantic/episodic(agent-writable) split. Rule layer (agents/*.md) = procedural = human-edited only; MEMORY.md/vault = agent-writable. 'Convert reliable rule to hook' aligns with the harness's existing advisory-hook model.
- **End-of-run wrap-up + start-of-run read**(来源:learnings.md practice — orchestrator.dev/blog/2026-04-06 ; augmentcode.com):Mandatory final step writes 'what to avoid / what to change / what I'd do differently'; next run reads it first; compounds over time. → 适用:DOWNGRADE to a RELIABILITY question: 'read at start' is reliable as a prompt instruction in the always-loaded rule file (works on all 5 agents). 'write at end' is the unreliable half — a Stop-style hook can reliably TRIGGER the write attempt (cc/droid native hook; kilo/opencode .mjs plugin), but the hook must stay advisory/fast, so it should NUDGE a skill rather than do LLM extraction itself.
- **Hybrid retrieval (keyword + vector), keyword-first**(来源:RAG retrieval tradeoff — aloknecessary.github.io/blogs/bm25_vs_vector_search ; ragaboutit.com):BM25 (cheap, explainable, exact) + vector (semantic, costly) merged/re-ranked beats either alone (~7% NDCG). → 适用:DOWNGRADE: take only the keyword half. grep/ripgrep over markdown = the explainable, zero-infra, fail-open retriever. Compensate for keyword's literal-match weakness via authoring discipline: structured headings, explicit topic tags, and an index file (cf. the existing MEMORY.md index of named notes) so grep targets stable keys.
