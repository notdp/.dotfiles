Boundary Decision Guard 原理说明

这份文档解释“我们自己做的防护”为什么存在、如何工作、哪些地方会帮助 coding agent、哪些地方会拖慢或误导 agent。它不是 Hook API 文档。

问题本质

AI coding agent 最危险的失败，不是报错，而是“替你决定了边界”。

典型表现：spec 没说拒绝输入，agent 加了 422；只要求接增量埋点，agent 接到了共享路径；只要求采样，agent 默认把 limit 当成全局。

防护目标

让边界决策从“隐形默认”变成“可见事实”。

不追求让 agent 永远不犯错，而是让它在动手前、动手后、收尾前留下可 review 的证据。

1. 什么叫 Boundary Decision

定义：凡是会改变“系统接受什么、拒绝什么、默认怎么填、走哪条路径、用哪个数据源、算到哪个指标、影响哪些 caller”的决定，都是 boundary decision。它们通常不在函数内部显得复杂，但会改变产品行为或运营结果。
类别例子为什么危险
input / rejection新增 status_code=422、新增参数校验把“可处理”变成“拒绝”，用户体验和 API contract 变了。
default / fallbackor []、fallback 到另一来源看起来更稳，实际可能掩盖数据缺失。
shared path改了被多个 caller 共用的函数一个场景正确，其他路径被污染。
metric routing新增 label、StatsStore route、metric name数据可能落到错误桶里，线上看不到真实变化。
schema contract改 response envelope、字段语义、response_model前后端、调用方、监控全部依赖这个边界。
data / samplingcanonical vs raw snapshot、global limit vs per-platform limit会直接改变召回、覆盖面和成本判断。
context surfacehook、prompt、capsule、CLAUDE.md、AGENTS.md这些会改变后续 agent 的输入和行为偏置。

2. 我们的防护不是一个 hook，而是一条流水线

1 识别风险

从 prompt 关键词判断是否是 wrapper、schema、metric、data source、prod、context surface 等高风险任务。

2 注入清单

通过 boundary capsule 告诉 agent：这些边界不能默默自决，要列事实或问用户。

3 写前检查

PreTool gate 看 transcript 里是否已有 Boundary decisions、Callers、Data source、Metric route 等证据。

4 写后扫描

PostTool scanner 从 diff 中找高信号边界变化，例如拒绝逻辑、默认值、schema、metric route。

5 收尾交代

Stop check 要求 assistant 自己写非空 Boundary decisions: manifest。

3. 每一层防护解决什么问题

防护层它解决的问题它不能解决的问题失败时怎么看
Boundary capsule 让 agent 在高风险任务开始时看到“不要自决边界”的 checklist。 模型可能忽略提示；关键词可能漏召回。 如果 transcript 里没出现 capsule，说明 prompt 没命中关键词或 hook 未生效。
PreTool gate 把“你先列事实了吗？”前移到写代码前。 当前默认 advisory，不是硬阻断；也不理解代码语义。 如果看到 advisory 但 agent 继续改，说明需要考虑 block 模式。
Boundary scanner 捕捉 diff 里的高信号边界变化。 只能扫模式，不能判断业务是否正确；宽泛规则会误报。 误报多就收窄 regex 或加排除；漏报就补高信号 fixture。
Stop manifest 让边界变化在最终交付前被显式列出。 当前 systemMessage 不保证 agent 自动补救。 如果你想强制闭环，升级为 Stop decision:block。

4. 为什么不直接上最强阻断

强阻断的好处

  • agent 必须处理，不能忽略。
  • 更接近“安全门”，适合 secrets、危险命令、生产写入。
  • Stop block 能强制 final 前补 manifest。

强阻断的代价

  • 误报会直接打断 flow。
  • agent 可能为了过门写模板化废话。
  • 边界判断本身需要语义，硬阻断太早可能降低效率。
当前 P1 取舍：先用 advisory 观察噪音和召回，再决定是否把某些路径升级为 block。换句话说，P1 是“可见化和测量”，P2 才是“强制执行”。

5. 对 coding agent 的正负影响

影响维度正面负面如何 review
质量 更容易发现 silent validation、metric path 污染、schema drift。 无法证明业务正确，只能暴露可疑点。 看 finding 是否能对应真实风险；不能对应就算噪音。
速度 高风险任务少返工。 普通任务可能多看一段 capsule 或 advisory。 统计一周：误触发次数 / 真有用次数。
上下文 把关键规则按需注入，而不是常驻 AGENTS.md。 PostTool finding 会进上下文，占 token 和注意力。 capsule 要短；scanner 只保留高信号。
行为偏置 鼓励先列 caller、contract、data source、metric route。 agent 可能过度保守，频繁问用户。 如果小改也频繁问，收窄 high-risk 关键词。

6. 当前状态评分

召回覆盖

覆盖 input/rejection/default/metric/schema/data/context 等主类，但仍靠 regex 和关键词。

误报控制

已排除 docs/html/scanner 自身等噪音，但新增规则仍需观察。

强制闭环

Stop 当前是 advisory，不是强制。若目标是必须补 manifest，需要 P2 block。

效率影响

大部分检查只在写文件或 stop 时运行，成本可控；最大风险来自上下文噪音。

7. Review 时只看这 5 个问题

  1. 这次任务是否属于高风险边界？ wrapper、schema、metric、data source、prod、context surface。
  2. agent 写代码前有没有列事实？ Callers、Contract cases、Data source、Metric route、Schema contract。
  3. diff scanner 命中是否是真风险？ 如果不是，应该收窄规则或加排除。
  4. 最终有没有 Boundary decisions manifest？ 必须是 assistant 自己写的非空块。
  5. 是否应该从 advisory 升级到 block？ 只有重复事故或 agent 持续忽略时升级。

8. Manifest 长什么样

Boundary decisions: - schema-contract: kept existing {success,data,error} envelope; no response shape change (src/api.py:42, evidence: existing client contract) - observability-routing: metric goes to creator.incremental.path, not shared path (src/metrics.py:18, evidence: synthetic emit verified) - sampling-boundary: --limit is per-platform, not global (cmd/run.py:55, evidence: user approved)

一句话总结：这套防护的原理是“不要指望 agent 自觉停下来问”，而是在高风险任务的入口、写入、写后、收尾四个时点，把隐形边界决策变成可见证据。
