基于当前仓库文件核对 面向日常使用

仓库 Hooks 日常工作地图

这页只回答一个问题：你每天使用 Droid 改代码时，哪些 hook 会介入，它们会提醒什么、阻断什么，以及哪些结果只是辅助判断。

先看结论 触发链路 逐项配置 脚本职责 注入内容 运行伪代码 反馈强度 边界机制 分类策略 日常场景 限制 文件索引
先记这句：这套 hooks 是一条工作流，不是单个拦截器。它先给 agent 上下文，再在写文件或执行命令前设护栏，写完后扫风险，最后在结束前检查验证证据。
第一层
入口提醒

Prompt 阶段按任务类型注入短规则；固定纪律放回 AGENTS.md。

第二层
写前护栏

编辑前看边界事实；执行命令前拦截高风险 shell。

第三层
写后扫描

编辑后扫描长任务合同、残留问题、边界变化。

第四层
收尾检查

结束前提醒缺验证、缺 UI 证据、缺 Boundary manifest。

日常判断规则：additionalContext、permissionDecision: deny、decision:block 更像 agent 必须处理的信号。当前默认的 systemMessage 更像提醒，不要把它当成稳定的 hard gate。

关键输出字段：additionalContext

它不是脚本。additionalContext 是 hook 返回 JSON 里的一个字段，意思是“把这段文字追加进 agent 当前上下文”。脚本负责生成它，平台负责把它塞回上下文。

生成场景输入来源生成脚本输出内容
UserPromptSubmithook stdin 里的用户 promptcontext_capsule.py --event prompt匹配到的 capsule 文本，例如 security、debug、boundary 规则。
PostToolUse当前 git diff 和 untracked 文件context_capsule.py --event post-tool 调用三个 scan_*.pyscanner findings 表格，例如 residue、operational、boundary scan 结果。
{ "hookSpecificOutput": { "hookEventName": "...", "additionalContext": "这里是要注入给 agent 的 Markdown 文本" } }

1. 触发链路

Prompt
你发需求

按关键词注入 security、operational、debug、UI、planning、boundary capsules。

PreTool
动作前

编辑走 boundary gate；Execute 走 command guard。

PostTool
编辑后

统一跑 3 个 scanner，把 findings 注回上下文。

Stop
结束前

检查验证证据、UI 视觉证据、Boundary decisions manifest。

2. 逐项配置：讲完一个，再讲下一个

1 / UserPromptSubmit

你提交需求后：按关键词挑选专项 capsule

时机：你发出一条新需求后、agent 正式处理前触发。命令是 scripts/hooks/context_capsule.py --event prompt，timeout 10s。

程序机制：脚本读取 hook stdin 里的 prompt 字段，用 CAPSULE_RULES 里的 regex 做召回。命中后读取 agents/context-capsules/*.md，最多拼到 2200 字符，再返回 additionalContext。

input = hook_json prompt = input.prompt capsules = [] if prompt has prod/auth/db/deploy words: capsules.add("security-gitops.md") if prompt has migrate/sync/backfill/dry-run words: capsules.add("operational-task.md") if prompt has bug/error/failed words: capsules.add("debug-task.md") if prompt has ui/css/react/figma words: capsules.add("ui-task.md") if prompt has plan/spec/architecture words: capsules.add("planning-task.md") if prompt has wrapper/schema/metric/hook words: capsules.add("boundary-decision.md") output additionalContext(join(capsules, capped_to_2200_chars))
用户 prompt 命中特征注入 capsule注入内容的作用
prod、deploy、ssh、db、secret、pushsecurity-gitops.md提醒先走安全或 GitOps 路径。
migration、sync、backfill、dry-run、batchoperational-task.md提醒长任务要有 dry-run、进度、恢复、apply 安全。
bug、fail、traceback、报错debug-task.md提醒先复现、看日志、建立假设。
ui、css、react、figma、overflowui-task.md提醒视觉验证、截图、viewport、overflow。
方案、架构、plan、optionsplanning-task.md提醒先规划再动手。
wrapper、schema、metric、data source、hook、AGENTS.mdboundary-decision.md提醒先列边界事实和 contract cases。

6 个 capsule 注入后分别长这样：

security-gitops.md

# Security / GitOps Capsule This prompt touches auth, permissions, secrets, deployment, production, remote machines, databases, or external state. - Load `/guard-secure` for auth/data/network/supply-chain risk. - Load `/guard-gitops` before remote, deploy, DB, secret, runtime, third-party, push, or release side effects. - Read-only diagnosis is allowed when scoped to observation and reported back. - Remote writes, deploys, DB writes, push/release, or apply commands need explicit scope, rollback, and validation evidence before proceeding. - Broad destructive cleanup, force-push, destroy, delete, or schema-destructive DB commands remain blocked. - Before delivery, report what changed, where it is declared in git, and how rollback works.

operational-task.md

# Operational Task Capsule This prompt looks like long-running, batch, data-changing, migration, or complex CLI work. Load `/dev-operational-task` before implementation. Check the contract before coding and before delivery: - Efficiency: bounded concurrency, batch size, backpressure, rate limit. - Resumability: cursor/checkpoint/state file/idempotency key; interrupted work can continue. - Observability: phase, current/total, percentage, rate, ETA, heartbeat, recent success/failure samples. - Robustness: retry/backoff, timeout, partial failure set, safe rerun behavior. - CLI UX: safe defaults, presets or interactive wizard; each choice explains impact and prints the underlying copyable command. - Dry-run data accuracy: planned counts, samples, diff/aggregation, invariants, failed examples. - Apply safety: `--apply` needs explicit confirmation and remote/db/destructive side effects route to `/guard-gitops`.

debug-task.md

# Debug Task Capsule This prompt looks like a bug, error, flaky behavior, failing test, traceback, or incident. - Load `/dev-debug` before changing code. - First build a feedback loop: failing test, CLI fixture, HTTP check, browser check, or trace replay. - Do not guess and patch. State hypotheses and verify one at a time. - After two failed hypotheses, switch to `/think-unstuck` with a handoff. - Final claim needs original repro evidence plus regression validation.

ui-task.md

# UI Task Capsule This prompt looks like UI, CSS, React/Vue/Svelte, page, screenshot, Figma, or visual work. - Load `/fe-ui-design` for implementation and `/fe-ui-visual-iterate` when comparing visuals. - Do not claim UI is done with tests only. - Capture real screenshots after changes, including mobile and desktop when relevant. - Check horizontal overflow, text overflow, interaction states, and design-token adherence. - If a Figma link is involved, use Figma MCP when available or suggest connecting it.

planning-task.md

# Planning Task Capsule This prompt asks for approach, architecture, options, phase plan, or a large change. - Load `/think-plan` and stay in spec mode until the plan is approved. - Use `/think-research` first when technical facts or options are uncertain. - Separate goals, non-goals, constraints, risks, validation, and rollback. - If there are equally strong options, ask the user to choose before finalizing. - Do not start implementation from an unapproved design.

boundary-decision.md

# Boundary-Decision Capsule This task looks like it may change a service, wrapper, adapter, schema, metric, data source, production path, or agent context surface. Before acting, surface facts or ask the user when you might change: 1. spec-external input validation, rejection, or 4xx/5xx behavior 2. skip, truncate, fallback, silent catch, retry, backoff, or sleep behavior 3. default values, limits, cost caps, concurrency caps, or sampling boundaries 4. shared code paths used by multiple callers 5. API request/response schema, envelope, or field semantics 6. canonical/raw data source, snapshot source, limit scope, or platform coverage 7. metric name, label, route, StatsStore bucket, or observability ownership 8. production writes, database changes, deployments, or external side effects 9. hooks, prompts, capsules, CLAUDE.md, AGENTS.md, or anything entering model context Use this manifest when a boundary changed or was considered: Boundary decisions: - <type>: <description> (file:line, evidence: <why allowed or user-approved>)

日常效果：这一步是“软路由”。它不会拦截你，也不会证明分类一定对；它只是把可能相关的短规则塞进上下文。

2 / PreToolUse: ApplyPatch | Create | Edit

写文件前：Boundary gate 先看你是不是在改边界

时机：agent 准备调用写文件工具前触发。Droid 配置匹配 ApplyPatch|Create|Edit；Claude 配置里对应 Edit|Write|MultiEdit。

命令：scripts/hooks/boundary_gate.py，timeout 10s。

程序机制：脚本读取 hook stdin 的 tool name，再读取 transcript。它只取最近一次用户 prompt，按命中词分类成 schema、metric、data source、shared path、context surface、limit/default/fallback、prod side effect 等风险；再只看这条 prompt 之后的对话里有没有结构化 Boundary facts: 或用户明确批准。旧历史里的 facts、assistant 自己写的“请确认”都不算。

换成人话：它不是判断代码对错，而是在 agent 写文件前问一句：“这次是不是会顺手改接口、默认值、fallback、指标、数据源或 prompt/hook 注入面？”如果会，而当前这轮对话还没把 caller、contract case、data source、metric route 这些边界说清楚，就先提醒 agent 按固定格式补齐事实再动手。

input = hook_json tool = input.tool_name records = read_transcript_records(input.transcript_path) prompt_index, prompt = last_user_prompt(records) if tool is not an edit tool: output suppress risks = classify prompt into schema, metric, data-source, shared-path, context-surface, limit/default/fallback, operational-side-effect window = records after prompt_index facts = structured Boundary facts fields in window missing = required fields for risks that facts do not satisfy if missing: if env.BOUNDARY_GATE_MODE == "block": output decision:block with missing fields else: output systemMessage advisory with missing fields else: output suppress

agent 应该写的固定格式：

Boundary facts: - Risk types: schema-contract, shared-path - Callers: creator search API - Contract cases: keep accept/reject/schema behavior - Data source: not applicable - Metric route: not applicable - Schema contract: response envelope unchanged - User approval: not requested

提示长什么样：

Boundary gate advisory: high-risk boundary prompt detected before edit, but no boundary facts were found. Missing: Schema contract or Contract cases. First list a structured `Boundary facts:` block or ask the user to confirm the boundary.

输出强度：默认是 systemMessage，属于 advisory；如果设置 BOUNDARY_GATE_MODE=block，会返回 decision:block。

日常效果：它不判断代码对错，只逼 agent 在动手前把边界讲清楚，避免“顺手改了 schema / 默认值 / 指标路由 / 上下文注入面”。

3 / PreToolUse: Execute

执行命令前：Command guard 给命令分三档

时机：agent 准备调用 shell/命令工具前触发。Droid matcher 是 Execute；Claude 对应 Bash。

命令：scripts/hooks/command_guard.py，timeout 10s。

程序机制：脚本从 hook stdin 的 tool_input.command 取 shell 命令，用 shlex 切词，再按命令段判断风险。现在分三档：明确只读和 dry-run 直接放行；发布、DB 写、远端写、git push、kubectl apply、terraform apply 等常见副作用命令返回 systemMessage 提醒；宽作用域清理、force push、destroy、delete、schema-destructive DB 命令继续 hard deny。

input = hook_json command = input.tool_input.command segments = shell_split(command) decision = allow for segment in segments: if segment is clearly read-only or dry-run: continue if segment is catastrophic: decision = deny break if segment has intended side effect: decision = warn if decision == deny: output permissionDecision: deny elif decision == warn: output systemMessage advisory else: output suppress
档位例子输出
allowgit status、只读 ssh、scp remote local、rsync --dry-run、SELECTsuppressOutput
warngit push、kubectl apply、terraform apply、远端写、DB 写systemMessage
denygit push --force、git clean -fdx、rm -rf /、terraform destroy、kubectl delete、DROP TABLEpermissionDecision: deny

warn 提示长什么样：

{ "systemMessage": "Command guard advisory: git push modifies remote repository state; route through /guard-gitops thinking and confirm branch, remote, CI impact, and rollback before proceeding." }

deny 提示长什么样：

{ "hookSpecificOutput": { "hookEventName": "PreToolUse", "permissionDecision": "deny", "permissionDecisionReason": "git force push can overwrite remote history; do not run automatically." }, "suppressOutput": true }

输出强度：allow 只静默放行；warn 是 systemMessage advisory，提醒先确认 scope / rollback / validation；deny 才是硬拦截，命中后 agent 必须换路径或请求明确授权。

4 / PostToolUse: ApplyPatch | Create | Edit

写文件后：自动复查刚改出的 diff

一句话：agent 每次写完文件后，hook 会马上看一眼当前 diff，检查有没有常见收尾风险；如果有，就把提醒重新塞回 agent 上下文，让它先处理或解释，不要直接继续。

时机：agent 用写文件工具后触发。Droid matcher 是 ApplyPatch|Create|Edit；Claude 对应 Edit|Write|MultiEdit。

入口命令：scripts/hooks/context_capsule.py --event post-tool，timeout 15s。

关系：context_capsule.py 是这个 PostTool hook 的入口脚本，也是 orchestrator。它自己不负责判断每一种风险，而是读取当前 diff 后，依次调用下面三个 scanner 脚本；scanner 只返回 findings，最后由 context_capsule.py 合并成一段 additionalContext 注入回 agent。

和 UserPromptSubmit 的区别：两者都用 hookSpecificOutput.additionalContext 这个同一个 hook 输出通道，所以都会进入 agent 上下文；但内容来源不同。--event prompt 注入的是按用户需求匹配到的 capsule 文本，--event post-tool 注入的是写文件后扫描当前 diff 得到的 findings 表格。

角色脚本负责什么
入口 / 调度scripts/hooks/context_capsule.py --event post-tool被 hook 直接执行；收集 diff；调用三个 scanner；合并输出。
worker scannerscan_operational_task_contract.py看 dry-run / apply / batch / migration 这类长任务线索。
worker scannerscan_diff_residue.py看临时代码、调试残留、疑似 secret 或危险占位。
worker scannerscan_boundary_decisions.py看新增 reject/default/fallback/schema/metric/early-exit 等边界分支。

伪代码归属：下面不是某个 scanner 的内部逻辑，而是 context_capsule.py --event post-tool 这个入口脚本的调度流程。

context_capsule.py --event post-tool: diff = staged_diff + unstaged_diff + untracked_files findings = [] for scanner in [ scan_operational_task_contract.py, scan_diff_residue.py, scan_boundary_decisions.py ]: findings += run(scanner, diff) if findings is empty: return suppress return hookSpecificOutput.additionalContext(markdown_table(findings))

markdown_table(findings) 渲染后长什么样：hook 实际注入的是 Markdown 文本；这里直接按表格样式展示它的内容。真实输出是一条 finding 一行，可能有多行；下面只是多行示例。

类型文件行片段建议
implicit-branch scripts/example.py 42 if not value: 新增边界分支后接 early exit；确认这个行为在 spec 内。
debug-residue scripts/example.py 87 temporary_probe(payload) 疑似调试残留；确认是否需要移除或改成正式日志。
operational-gap scripts/backfill.py 120 --apply 涉及 apply 路径；补充 dry-run 数据证据、进度和回滚说明。

总结：示例命中 3 条；真实数量取决于当前 diff。

你会看到什么：类似你刚才贴的 Boundary Decision Scan 表格，里面有类型、文件、行号、片段和建议。

不会做什么：它不会自动改代码，也不会 hard block；它只是写后复查，把可疑点重新提醒给 agent。

5 / Stop

结束前：Stop check 看有没有验证证据和收尾缺口

时机：agent 准备结束回复前触发。命令是 scripts/hooks/stop_check.py，timeout 10s。

程序机制：脚本读取当前 git status --short 和 transcript。只要发现代码文件有改动，就只检查最近一次用户请求及其之后的窗口里有没有测试、lint、typecheck、verify_skills.py 等验证证据；如果 UI 文件有改动，还会额外找 screenshot、visual、overflow、playwright 等证据；如果这个窗口里出现过 boundary scanner finding，还会找 assistant-authored Boundary decisions: manifest。旧历史里的验证和 manifest 不再算当前任务证据。

input = hook_json files = git_status_short() records = read_transcript_records(input.transcript_path) window = records from latest user prompt onward code_files = files with code suffix if no code_files: output suppress problems = [] if window has no validation evidence: problems.add("missing validation evidence") if changed files include UI code and window has no visual evidence: problems.add("missing visual evidence") if window has boundary scanner finding and no assistant-authored Boundary decisions block in window: problems.add("missing Boundary decisions manifest") if problems: output systemMessage(join(problems)) else: output suppress

提示长什么样：

Stop check advisory: code changes exist but no validation evidence was found; run /guard-verify before claiming completion. Changed files: scripts/...

输出强度：当前是 systemMessage advisory，不是 hard gate。

日常效果：它在最终回复前提醒“别直接说完成”。但如果 agent 忽略 systemMessage，仍可能继续输出，所以真正交付还要看验证证据。

scripts/hooks/context_capsule.py hook stdin JSON、prompt、repo diff hookSpecificOutput.additionalContext 或 suppressOutput 上下文注入 把你的 prompt 和当前 diff 转成短上下文；具体判断见前面的对应配置章节。

4. 注入内容如何 review

本质：context_capsule.py 做的是 prompt routing。它不改代码，也不调用模型；它只根据用户 prompt 里的关键词，读取 agents/context-capsules/*.md，再把命中的文件内容拼成 additionalContext 注入给 agent。
Capsule 文件何时注入注入后要求 agent 做什么Review 重点
security-gitops.md prompt 提到 prod、deploy、ssh、push、auth、DB、secret 等词。 要求安全审查；远端、部署、DB、secret、push、release 先走 GitOps。 关键词不能太宽，否则日常问题会被安全流程打断。
operational-task.md prompt 提到 migration、sync、backfill、batch、dry-run、apply 等词。 要求长任务具备效率、可恢复、可观测、dry-run 数据证据、apply 安全确认。 适合数据/批处理任务；普通脚本不要被误判成长任务。
debug-task.md prompt 提到 bug、error、failed、traceback、异常、复现等词。 要求先建立复现反馈环，再按假设逐个验证，不要猜改。 这类注入通常低风险，但要避免把普通咨询误判成调试任务。
ui-task.md prompt 提到 UI、CSS、React、页面、截图、Figma、overflow 等词。 要求 UI 任务截图验证，检查溢出、状态、设计 token；Figma 走 MCP。 如果项目里大量文件名含 UI 词，注意是否频繁误触发。
planning-task.md prompt 提到 方案、计划、架构、approach、spec 等词。 要求先计划，方案未批准前不实现；多方案时先让用户选择。 这个会改变 agent 节奏，关键词过宽会让小问题也进入规划模式。
boundary-decision.md prompt 提到 wrapper、schema、metric、data source、hook、AGENTS 等词。 要求先暴露 validation、fallback、shared path、schema、metric、data source、context surface 等边界。 这是最容易误报也最有价值的一类；重点看是否真的改变系统边界。
Review 方法：先看 CAPSULE_RULES 的关键词是否过宽，再看对应 capsule 是否只保留“必须改变 agent 行为”的短规则。不要把解释性长文塞进 capsule；长解释应该放 docs。
python3 scripts/hooks/context_capsule.py \ --event prompt \ --preview \ --prompt-text "给我通过 html 清晰展示当前仓库 hooks"

这个 preview 命令只打印 Markdown 报告，不走 hook JSON。报告会列出每个 capsule 的 matched / not matched、heading、regex 规则和最终上下文长度。

6. 三个写后 scanner 的明细

和第 4 节的关系：第 4 节讲“写完文件后会触发一次复查”；本节只展开那次复查里的三个 scanner 分别抓什么、哪些情况可能是误报。
Operational

scan_operational_task_contract.py

识别批处理、迁移、同步、回填、长任务 CLI 等 operational-looking diff。

检查：进度可见、可恢复、dry-run 数据证据、失败可诊断、有界并发、apply 安全确认。

常见误报：普通小脚本里出现 dry-run、batch 这类词，但并没有真实长任务或数据变更。

Residue

scan_diff_residue.py

扫描新增 diff 中的临时残留和疑似凭据。

检查：待办标记、调试前缀、临时输出语句、断点调用、疑似凭据、临时变量名。

常见误报：测试 fixture、文档示例或刻意保留的 diagnostic 输出。

Boundary

scan_boundary_decisions.py

扫描可能改变系统边界的新增代码。

检查：4xx/5xx rejection、assert guard、默认值、retry/backoff、schema contract、metric route、silent catch、early exit。

常见误报：工具脚本里的输入校验、测试用例里的边界样例、以及已经存在但因为 untracked 文件被反复扫描到的代码。

7. Boundary 机制

为什么重要：hook、prompt、capsule、schema、metric、data source 这类改动会改变后续 agent 或调用方的边界。当前设计把它拆成四层：Prompt 注入 capsule、PreTool 写前提醒、PostTool diff 扫描、Stop 检查 manifest。
UserPromptSubmit -> context_capsule.py 匹配 boundary 关键词 -> 注入 agents/context-capsules/boundary-decision.md PreToolUse: ApplyPatch|Create|Edit -> hook_boundary_gate.py -> 默认 systemMessage；BOUNDARY_GATE_MODE=block 时 decision:block PostToolUse: ApplyPatch|Create|Edit -> scan_boundary_decisions.py --hook -> 命中后追加 Boundary decision scan found... Stop -> hook_stop_check.py -> 若 transcript 里有 boundary finding 但无 assistant-authored Boundary decisions manifest，则提醒

8. 规则还是便宜模型

建议：不要把所有判断都交给模型。更稳的设计是 hybrid：硬安全用规则，软路由用便宜模型辅助，最终注入仍要有白名单、短上下文预算和日志。
方案适合做什么优点风险结论
纯 regex 命令拦截、明显关键词、低成本 capsule routing。 可审计、稳定、快、无额外模型成本。 语义弱；容易误报，也可能漏掉换一种说法的关键任务。 适合 hard guard 和第一版实现。
便宜模型分类 判断 prompt 是否真的属于 debug / planning / boundary / operational。 能理解语义，减少“词命中但任务不相关”的脏 context。 有延迟、成本、不可完全复现；分类错了仍会污染上下文或漏掉提醒。 适合 advisory routing，不适合单独做安全硬门。
Hybrid 规则先召回候选，模型只在候选多或不确定时裁剪。 保留可审计性，同时降低误注入；可以记录分类结果方便回放。 实现复杂度更高，需要 timeout、fallback、日志和测试样本。 推荐。先给 capsule routing 用，不碰 command guard。
recommended_classifier_flow(prompt): candidates = regex_rules(prompt) if no candidates: return [] if candidates are high-confidence hard safety: return candidates model_result = cheap_model_classify( prompt, allowed_labels=["debug", "ui", "planning", "operational", "boundary", "security"], max_labels=2, require_reason=true ) selected = intersection_or_explained_override(candidates, model_result.labels) log(prompt_hash, candidates, selected, model_result.reason) return selected

11. 限制

限制影响日常处理方式
关键词和 regex 驱动 会有误报和漏报，尤其是 boundary 与 operational 类规则。 把 finding 当 review cue，不要当业务正确性证明。
Stop 当前是 advisory 它能提醒，但不是默认 hard gate。 如果你想强制闭环，下一步要改成 decision:block。
PostTool scanner 会进上下文 误报会占 token 和注意力。 发现噪音后优先收窄规则或加排除。
Scanner 看 diff，不懂完整业务语义 能发现风险形态，不能证明需求验收通过。 最终仍要跑测试、验证命令，必要时做人工可观察验收。

备注：默认不安装 SessionStart / PreCompact 自定义 hook；固定纪律放在 agents/AGENTS.md，scripts/hooks/context_state.py 只保留作 legacy/manual 工具。
