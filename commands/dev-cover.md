# Test Coverage Sweep（分层覆盖梳理）

把"列出这次都改了什么 → 怎么测 → 补测覆盖"这套手动操作固化下来。**锚点是 diff/commit，不是对话记忆**——对话会丢、会漏，diff 不会。

## 0. 适用场景

- AI 排查问题或开发特性后，怀疑测试只停在底层、没覆盖功能点
- 要防止后续改动破坏本次特性（回归防护）
- 提交或交付前，确认每个功能点有对应层级的测试

## 1. 锚定改动（用 diff，不用记忆）

```bash
git status --short --untracked-files=all      # 未提交改动（含新目录里的文件）
git diff --stat                                # 行级范围
# 指定范围：git diff --stat <base>..<head>
```

列出**源文件改动**，排除纯文档/配置/测试文件自身。每个源文件改动对应一个或多个"功能点"。

## 2. 逐项映射层级

层级口径复用 `/dev-tdd` 的"测试分层"（功能点 / 集成 / 单元），不另立一套。对每个功能点：

- 它的可观察面在哪一层 → 那一层是否已有测试锁住？
- 修 bug 的：是否有功能点级或集成级的**复现测试**？只有单元测试 = 这个 bug 没被覆盖。

## 3. 输出缺口表

| 功能点 | 改动文件 | 应锁层级 | 现有测试 | 缺口 | 补测建议 |
|---|---|---|---|---|---|
| <可观察行为> | <file> | 功能点/集成/单元 | 有 / 无 / 仅单元 | <缺哪层> | <该测什么> |

## 4. 补测

对每个缺口走 `/dev-tdd`（先红后绿）：缺口若在功能点级，先写功能点级失败测试再补实现。不是每层都补——每个功能点锁在最能防回归的那一层，其余层显式说明为何豁免（重构/胶水/纯配置）。

## 完成判据

- 每个源文件改动的功能点都进了缺口表，且每行有"应锁层级 + 现有 + 缺口"裁定
- 每个缺口要么补测（红→绿），要么显式写出豁免理由
- 不用"新增单元测试"冒充功能点覆盖；不靠对话记忆重建改动清单
