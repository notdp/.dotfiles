# Readable Final Answer

合并两类能力：

- **Mode A 内容重写**：把表述太密、太绕、难读的段落改顺
- **Mode B 体裁规范**：约束 agent 自身最终输出 / PR 描述 / 过程播报的形式

进入 skill 时先判断目标属于哪个 mode；两者可叠加（先重写再套体裁）。

核心方法论：**降低读者手里的未知概念数量，并按读者能吸收的顺序逐步披露信息**。可读性优先是结构问题，其次才是措辞问题。

---

## 0. 认知负担模型

写作前先按这 4 条判断读者负担：

1. **先地图，后细节**：读者需要先知道整体位置，再逐层进入细节；不要把背景、例外、trade-off、实现细节一次性摊开。
2. **先熟悉，后陌生**：人更容易从已知经验迁移到新概念；先接住用户已熟悉的对象、痛点或感受，再引入术语和抽象。
3. **控制概念预算**：关键结论、风险、行动项默认控制在 3-7 项；超过就分组、分节、折叠或拆成多轮。这里的 3-7 是写作启发，不是严格心理学定律。
4. **能画就画，但不滥画**：流程、状态、依赖、对比、时间线、风险分布优先考虑图表；短结论、单文件总结、验证摘要不要为了形式感做图。

## Mode A: 内容重写

### 1. 明确输入

如果 `$ARGUMENTS` 不足以判断上下文，先补齐：

- 原文是什么？
- 主要读者是谁？（工程师 / 跨职能 / 新人 / 混合受众）
- 这段内容的用途是什么？（架构说明 / 指标解读 / 错误信息 / 通用说明）
- 哪些事实、数字、术语、结论必须原样保留？
- 用户要的是"直接重写"，还是"先诊断再重写"？

如果用户只给了目标、没给原文，先让用户提供原文，不要凭空生成"重写版"。

### 2. 诊断可读性问题

在动笔前，先判断原文的问题属于哪几类：

| 维度 | 典型症状 | 处理动作 |
|------|----------|---------|
| 信息顺序 | 结论埋得太深、背景先于答案 | 改成倒金字塔，先给结论 |
| 句子负担 | 句子过长、一个句子塞多层意思 | 拆句，一句一事 |
| 段落负担 | 一段混多个主题 | 拆段，一段一主题 |
| 术语负担 | 黑话多、缩写未解释 | 首次出现时解释 |
| 抽象负担 | 只说原则，不说对象/条件/动作 | 补充主语、动作、结果 |
| 数字负担 | 只有数字，没有语义和上下文 | 解释基线、变化、影响 |
| 结构负担 | 没有层级、列表、标题 | 改成可扫描结构 |
| 迁移负担 | 一上来讲陌生概念，缺少读者熟悉锚点 | 从已知对象、感受、类比或例子进入 |
| 概念负担 | 同屏概念、风险、行动项太多 | 控制 3-7 项，超出就分组或拆节 |
| 图示机会 | 文字在描述流程、依赖、状态、对比 | 改成图、表、流程、矩阵或时间线 |
| AI 味（见下） | 千篇一律的句式、空话填充 | 按 AI 写作信号 checklist 逐条修 |

#### AI 写作信号 quick checklist

诊断时同时扫这些信号；命中 ≥ 2 条说明文本带 AI 味，需要重写：

- **em dash 过载**：每段 ≥ 2 个 em dash，或用 em dash 串接整段
- **rule of three**：每段都用三件事并列结构（"不是 X，不是 Y，而是 Z"）
- **negative parallelism**：反复用 "not just X, but Y" / "不仅是…更是…" 套娃
- **AI 词汇命中**：delve / leverage / navigate / unlock / embark / tapestry / robust / seamless / cutting-edge / 赋能 / 助力 / 打造 / 重塑 / 生态
- **千篇一律的句长**：连续 5 句以上句长接近，节奏单调
- **无"我"视角**：通篇第三人称中性叙述，无观点、无立场、无情感色彩
- **通用 "-ing" 转折**：每段开头 "Building on..." / "Looking at..." / "随着…的不断深化"
- **空泛承诺**：`prevent` / `ensure` / `guarantee` / `eliminate` / `seamlessly` 出现而无证据
- **过度对仗**：每段两两对仗，结构对称到刻意
- **元叙述堆叠**："In this article, we will explore..." / "本文将带你深入了解…"

修复方向：句长变化、加观点、削掉空话、用具体对象代替抽象、保留必要不确定性。

### 3. 重写原则

#### 信息顺序

- 结论先行：先回答“所以呢”，再解释背景
- 金字塔展开：`结论 → 地图/分组 → 关键点 → 证据/细节 → 例外`
- 渐进披露：先隐藏低频细节；必要时用“详情 / 例外 / 证据”分节下钻
- 先熟悉后陌生：先用用户已有问题、已有系统、常见体验做锚点，再引入新术语
- 一次只引入一层抽象；不要把概念定义、架构关系、trade-off、实现步骤混在同一段

#### 结构

- 先扫描，后阅读：标题本身要传达结论
- 先结论，后背景：倒金字塔
- 先总览，后细节：渐进展开
- 一段一主题；长段拆开
- 3 项以上并列信息优先用列表
- 比较信息优先用表格
- 超过 7 个同级点时，先分组，再列点；不要让读者同时抓太多“球”

#### 语言

- 尽量用主动语态
- 一句只说一件事
- 用最简单但不失真的词
- 避免双重否定、空泛形容词、跳跃式代词指代
- 术语首次出现时给出解释；必要时保留"原词 + 白话解释"

#### 说人话红线（口语优先，服从保真）

来自 `refs/lijigang/ljg-skills` 的 ljg-plain。上面"语言"是底线，这几条是把"AI 腔"改成"人话"的更硬尺子，按顺序即优先级逐条过。全部服从下方 §5 保真规则——**口语管"怎么说"，保真管"说什么"，冲突时准确赢**。

1. **口语检验（最高法则）**：把句子读出声——你会这样跟一个聪明朋友说吗？不会就改到会。只砍机械连词（"此外""值得注意的是"）；"但是""所以"是思维转弯的声音，保留。
2. **零术语**：能让外行复述才算过。术语必须出现时，先用大白话把意思落地，再顺带提术语名（"把请求攒一批再发——也就是 batching"）。
3. **短词优先**：能两个字别用四个字。"进行分析"→"看"，"做出决策"→"定"。大词不显得聪明，只让人读着累。
4. **具体压抽象**：名词看得见，动词有力气，形容词能砍就砍。"有人觉得情况不太好"→"张三说项目要黄了"。
5. **诚实于不确定**：想不清就说想不清。"大概 70%" 比"可能"诚实——别拿模糊词盖住其实知道的量级。
6. **信任读者**：说一遍够了，跳过软化、辩解、手把手；不"自古以来"式铺垫，第一句就给读下去的理由。
7. **查译感（中文特有）**：动宾搭配中文天然吗？"进行一个优化"不天然 →"优化一下"。译腔多来自硬套英文结构。

#### 数字

- 不只报数字，要解释数字代表什么
- 给出基线、阈值、变化方向、异常程度
- 非必要场景可适度取整，避免伪精确

#### 图表

- 一图只回答一个问题；图后必须跟一句“这说明什么”
- 流程用流程图，关系用结构图，对比用表格/矩阵，时间演进用 timeline，风险分布用热力图或状态表
- 图表复杂到需要读者反复查图例时，拆图或改回分节文字
- 图表不能替代结论；图表只负责降低关系、顺序、差异的理解成本

### 4. 输出格式

默认按下面顺序输出：

```md
## TL;DR
[一句话概括]

## Rewritten
[分层重写后的正文]

## Terms / Examples
[仅在需要时提供]

## What Changed
- [仅在用户要求解释改动时输出]
```

对不同内容类型做额外约束：

- **架构说明**：先给地图，再讲层次、链路、决策
- **指标内容**：先给结论，再解释变化、影响、下一步
- **错误/告警**：先说发生了什么、对用户意味着什么、下一步做什么

### 5. 保真规则

- 不新增用户未提供的事实
- 不偷偷改变结论、数字、单位、范围、因果关系
- 不为了"更顺"而抹掉限制条件、例外情况、风险提示
- 如果原文本身含糊到无法安全重写，先指出歧义，再请用户澄清

### 6. 自检

- [ ] 读者扫一眼标题和列表，能否抓住主结论？
- [ ] 是否先给结论，再给解释？
- [ ] 是否先给地图，再逐层暴露细节？
- [ ] 是否从读者熟悉的概念、感受或例子进入？
- [ ] 是否一段一主题？
- [ ] 同一屏关键概念、风险、行动项是否控制在 3-7 项或已分组？
- [ ] 流程、依赖、对比、时间线是否已考虑表格或图示？
- [ ] 图表是否一图一问题，并紧跟一句解释？
- [ ] 是否删掉了多余抽象词和双重否定？
- [ ] 关键句读出声，像跟聪明朋友说话，而非像 AI 念稿（口语检验）？
- [ ] 术语都先用大白话落地了，没有裸术语？
- [ ] 是否保留了所有关键事实、数字、术语和约束？
- [ ] 是否把需要上下文的数字解释清楚了？

### 7. Mode A Gotchas

- 口语检验只管"怎么说"，保真规则管"说什么"——两者不冲突；技术准确性永远优先，别为顺口抹掉限制条件或改动结论
- 不要把原文没有的判断、承诺、因果强行补进去
- 不要只做同义词替换；真正的问题通常在结构，不在词汇
- 原文已经很短时，重点是重排顺序，不是强行扩写

## Mode B: 最终答案体裁规范（来自 codex 收编）

适用于 agent 输出最终答案、PR 描述、release notes、用户文档等场景。

### 长度与形态

- 简单 / 单文件任务：1–2 段 prose，可加一行验证语
- 复杂任务：分节，但整体 ≤ 50–70 行
- 不以 "If you want…" 句式收尾
- 不写"我做了 X 而不是 Y"这种自夸式对照
- 单/双文件改动用 prose 收尾，不默认上 bullet

### HTML Companion

最终答案默认用 Markdown，保持可复制、可审查、可 diff。HTML companion 是同步交付物，不是事后口头建议。

以下任一条件命中时，触发 HTML companion artifact：

- 用户显式要求 HTML / companion / artifact / 可视化版本。
- 用户在一段已有最终答复后手动触发本 skill，且原答复包含多层标题、表格、指标、路线判断、风险判断或行动计划。
- 内容属于长报告、PR explainer、跨职能分享、领导层简报、复杂视觉说明或交互式审阅。

以下情况不要生成 HTML 文件，但必须在最终答复末尾用一句话说明未生成原因：

- 短最终答复。
- 单文件改动总结。
- 验证摘要。
- 用户只要求改写一小段文本，且没有可视化或分享需求。

HTML companion 硬约束：

- Markdown 仍是最终结论和事实边界的 SSOT。
- HTML 只负责 presentation：层级、图示、注释、交互、分享。
- 一旦触发 HTML companion，不要只口头说明或只给 Markdown；若当前上下文禁止写文件或仍在审批/spec mode，先声明批准后生成 HTML；允许写文件后通过 `/readable-html-artifact` 生成 `.html` 文件。
- HTML 必须引用对应 Markdown source 或对话结论。
- HTML 生成必须委托 `/readable-html-artifact` 或其 `scripts/render_html_artifact.py` renderer；本 skill 不手写、不粘贴完整 HTML。
- HTML 必须结构化：TL;DR、目录 / 导航、关键结论卡片、证据区、风险区、行动区分层清楚，避免把 Markdown 原文整段搬进页面。
- 复杂内容优先图表化：能用矩阵、时间线、流程图、状态图、对比柱状图、风险热力图、指标卡、证据表说明的内容，不只用段落解释；图表数量以降低理解成本为准，不为凑数量堆图。
- 批准后生成 HTML 时，必须立即打开浏览器预览（macOS 用 `open <file.html>`，其他环境用等价方式）；如果当前环境无法打开 GUI / 浏览器，报告 HTML 路径和未打开原因。
- 最终回复必须同步给出 Markdown source、HTML 文件路径和 renderer 验证摘要；不要返回完整 HTML，也不要只说"可以生成"。

参考：
- `refs/trq212/unreasonable-effectiveness-of-html.md`
- `refs/danielmiessler/text-is-thought-holy.md`

### Markdown 规则

- GitHub-flavored Markdown
- 列表平铺，避免嵌套；需要层级时拆节或换行 + 冒号
- 数字列表用 `1. 2. 3.`，不用 `1)`
- header 用 `**Title Case**`（1–3 词），不空行
- 命令 / 路径 / env / code id 用 backtick
- 多行代码用 fenced code block，尽量带 info string

### 文件链接

- 形式：`[label](/abs/path:line)`
- 路径含空格：`[label](</abs/path/My File.md:3>)`
- 不嵌 backtick、不用 `file://` / `vscode://` / `https://`
- 不写行号范围
- 同一文件多次出现时合并指代，避免重复贴路径

## Mode C: 过程播报（intermediary updates）

适用于 agent 在工作中、edit 前、长流程中的中间状态发声。

### 节奏与风格

- 每次动作前先一句简短说明（在改什么 / 在读什么 / 为什么）
- 句式多样化，不要每条都用同一开头
- 单条 ≤ 2 句；只有需要长 plan 时才允许超过
- 探索（搜索 / 读文件）时持续 narrate 在收集什么、学到什么

### 内容约束

- 不夸自己的方案，不与"显而易见的差方案"作对比
- 不报告已经做完的琐碎步骤；只报对用户有信号的进展、决策、卡点

---

## 文风禁忌（适用全部三种 mode）

- **说人话红线同样适用 Mode B/C**：agent 自己的最终答案、PR 描述、过程播报也要过 Mode A §3「说人话红线」——口语检验、零术语、短词优先、查译感。这是"AI 不说人话"最常发作的地方。
- 不用 emoji（除非用户明确要求）
- 不用 em dash 作填充
- 不用动物比喻填充语：goblins / gremlins / raccoons / trolls / ogres / pigeons
- 不用自造隐喻：seam / cut / safe-cut 这类生造词
