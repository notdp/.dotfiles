# Learning loop

## 一条 learning note 至少回答 5 个问题

1. 这次处理的对象是什么？
2. 为什么之前会卡住或容易踩坑？
3. 最终采用了什么做法？
4. 哪部分可以复用到下次？
5. 还有哪些边界或后续项没解决？

## 写作约定

- 标题写问题域，不写空泛感想
- 关键术语保留英文原词，必要时补中文解释
- 面向团队共享时，摘要可以双语，但正文先求清晰再求花哨
- 经验必须附证据：测试、命令输出、代码路径、截图之一

## 优先沉淀的内容

- checklist
- decision rule
- reusable snippet
- anti-pattern
- verification recipe

## 规则措辞形式选择（Match the Form to the Failure）

把经验沉淀成给 agent / skill / hook 的规则时，**先分类它要防的失败类型，再选措辞形式**——能堵住一种失败的形式，对另一种会反效果。

| 基线失败 | 正确形式 | 错误形式 |
|---|---|---|
| 压力下明知故犯、跳过/违反规则 | 禁止式 + rationalization 表 + red flags（明知故犯靠堵借口） | 软措辞（"prefer…"/"考虑…"/"酌情"） |
| 照做了但**输出形状错**（prompt 臃肿、结论埋没、复述 spec） | 正向配方/契约：直接说输出 **是什么**——有哪些部分、什么顺序 | 禁止式清单（"别复述""别叙述"） |
| 漏掉本该产出的某个元素 | 结构式：在模板里放 REQUIRED 字段/槽位 | 模板附近的散文提醒 |
| 行为应随条件变化 | 挂在**可观测谓词**上的条件式（"若 brief 存在则引用它"） | 无条件规则 + 一堆例外豁免条款 |

- **禁止式为什么对"形状错"反效果**：当存在竞争性激励时，agent 会和"别做 X"讨价还价。实测中禁止式那组产出的不良内容明显多于配方组、甚至差于无指导对照组。配方没得谈：要么匹配规定的形状要么不匹配。
- **不要加 nuance 从句**："别做 X 除非有必要" 会重开谈判；真有例外就写成挂在可观测谓词上的独立条件式。
- **豁免条款不收敛 scope**："此限制不适用于代码块" 仍会压制代码块；某部分必须豁免就重构结构让规则够不到它。
- **先 micro-test 措辞再铺全量**：对自己的具体场景做"无指导对照 + 5+ 次重复"小测，variance 是指标，别默认伸手去拿禁止式。
