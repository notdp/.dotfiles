# Scope

与 think-plan 区别：think-scope 解决"用户要解决什么问题、agent 和用户对系统现状和改动范围的理解是否一致"，think-plan 解决"方案怎么设计、怎么拆步骤"。scope 清晰后再进 plan，不要跳过对齐直接写 spec。

## Decision Principles

- **问题先于方案**：用户给出的改动请求通常是手段而非目标。先搞清楚"要解决什么问题、现状为什么不可接受"，再讨论改什么、怎么改。搞清楚 why 是整个 scope 对齐的前提。
- think-scope 优化的是**需求理解的准确性**，不是需求的完备性或方案设计。
- 返工的主因有两个：(1) agent 没搞清楚用户要解决什么问题就开始讨论方案；(2) agent 对系统现状的心智模型与现实不符。
- 代码是验证问题理解的锚点：先锚定问题，再读代码验证问题是否如用户所述、找到问题的代码层根因，比跳过问题直接读代码更有效。
- 当"要解决什么问题、改什么、不改什么"都明确时停止，不要继续扩展成完整 spec。

## 核心循环

为什么这样循环：返工有两个主因——(1) agent 没搞清楚用户要解决什么问题就开始讨论方案（XY 问题）；(2) agent 在没有读代码的情况下建立了错误的系统心智模型。Problem → Ground → Model → Align 的顺序强制 agent 先锚定问题，再用代码证据验证问题理解，最后基于问题和代码上下文提出知情的问题。

```
1. Problem — 拆解输入，锚定问题（要解决什么、现状为什么不可接受）
2. Ground  — 基于已锚定的问题，定向读代码
3. Model   — 陈述"我理解的问题 + 系统现状 + 改动方向"
4. Align   — 交互式对齐（问题和代码知情的提问）
5. Scope   — 产出锚定后的 scope
```

### Phase 1: Problem（问题锚定）

在读代码之前，先搞清楚用户要解决什么问题。这个阶段不读代码——过早读代码会让 agent 被实现细节锚定，偏离问题本身。

1. **拆解输入**：把 `$ARGUMENTS` 和用户描述分为三层——problem（痛点/故障/限制）、intent（想达成什么）、solution hint（用户提出的改法）。用户输入几乎一定混着三层，skill 的第一步是拆开它们。
2. **追问 why**：如果用户只给了 solution hint 没有 problem，主动追问："这个改动要解决什么问题？现状哪里不可接受？"不要把用户给的改法直接当作需求。
3. **XY 问题检测**：判断用户给的是手段还是目标。例如用户说"帮我加个缓存"，先确认"是遇到了性能问题吗？瓶颈在哪里？"——用户要的可能不是缓存，而是解决慢的问题。
4. **锚定问题陈述**：用一句话陈述"问题是什么"，等用户确认后再进入 Ground。

快速通道：如果用户输入已包含明确的问题陈述（痛点 + 现状不可接受的原因），一句话确认即可，不展开追问。

### Phase 2: Ground（问题导向的代码分析）

基于 Phase 1 锚定的问题，定向分析代码。读代码的目的是**验证问题是否如用户所述、找到问题的代码层根因**，而非直接去看"需求涉及的功能区域"。

1. 找到问题发生的代码路径（入口文件、核心模块、数据流）
2. 验证用户描述的痛点在代码层是否成立，找到根因
3. 识别与问题相关的现有行为、边界条件、测试覆盖
4. 识别领域术语：代码中的命名、注释、文档是否与用户用词一致

分析范围聚焦在问题相关区域，不做仓库级全扫描（那是 think-map 的职责）。

使用工具而非猜测：Glob/Grep/Read 定位文件，不凭经验推断。大型项目可派发只读子任务并行分析不同模块。

### Phase 3: Model（陈述理解）

向用户陈述三件事，按以下顺序：

**问题定义**（基于 Phase 1 锚定 + Phase 2 代码验证）：
- 我理解你遇到的问题是什么
- 代码层面的根因是什么（带 `file:line` 证据）
- 问题的影响范围和严重程度

**系统现状**（基于代码证据）：
- 当前系统在这个区域是怎么工作的（关键路径、数据流、约束）
- 涉及哪些文件/模块/接口

**改动方向**（基于问题定义推导）：
- 基于以上问题和现状，改动方向是什么
- 改动会影响哪些现有行为
- 我的理解中有哪些不确定点

陈述必须带 `file:line` 或至少具体文件路径作为证据。未确认的推断标 `[推断]`。

### Phase 4: Align（交互式对齐）

基于 Phase 3 的陈述，与用户交互对齐。继承 think-refine 的提问纪律：

- 一次一个问题，不要一次列 5 个
- 每个问题提供 2-3 个候选选项 + 推荐，选项基于代码现实而非凭空构造
- 不替用户做决定

#### Ambiguity Score（停止条件）

四维评分，每维 0-1，越低越清晰：

| 维度 | 含义 | 为什么 |
|---|---|---|
| Problem Clarity | 要解决什么问题、现状为什么不可接受是否明确 | 防止解决错误问题（XY 问题） |
| System Model | agent 对系统现状的理解是否经过验证 | 防止基于错误心智模型做决策 |
| Boundary Clarity | 改什么 / 不改什么是否明确 | 防止 scope creep |
| Acceptance Clarity | 怎么算改好了 | 防止无法验收 |

总分 = 四维平均。 <= 0.20 放行，> 0.20 继续追问。

评分是 agent 自检工具，不是展示给用户的流程。哪维高就追问哪维。

#### 问题导向提问示例

先追问 why，再用代码证据追问 what/how：

| 层级 | 纯对话式提问 | 问题导向提问 |
|---|---|---|
| why | "你要加什么功能？" | "当前流程里哪个环节出了问题？是效率低、出错多、还是缺能力？" |
| why | "你要优化什么？" | "现在的性能在什么场景下不可接受？有观测数据吗？" |
| why | "你要怎么改这个接口？" | "这个接口的调用方遇到了什么困难？是返回数据不够、还是调用方式不便？" |
| what | "你要加什么类型的认证？" | "当前 `auth/middleware.ts:23` 用 JWT + cookie session。你是要在这套机制旁加 OAuth provider，还是替换掉？" |
| what | "数据存哪里？" | "`models/user.py:45` 的 User 表已有 email/password 字段，`services/cache.py` 用 Redis 做 session。新功能的数据要加字段还是新建表？" |
| what | "需要考虑什么边界情况？" | "`handlers/upload.ts:78` 当前限制 10MB，`config/limits.json` 可配置。你的需求是否需要调整这个限制？" |

#### 术语对齐

代码分析中发现的命名与用户用词不一致时，主动提出：

- "你说的'订单'在代码里叫 `Transaction`（`models/transaction.py`），后续用哪个词？"
- "代码里 `publish` 和 `release` 是两个不同操作（`services/publish.py` vs `services/release.py`），你说的'发布'指哪个？"

### Phase 5: Scope（产出）

对齐完成后产出 scope 文档（直接输出到对话中，不创建文件）：

```markdown
## Scope

### 问题定义
[要解决什么问题 + 现状为什么不可接受]

### 系统现状
[基于代码的关键路径描述，带 file:line 证据]

### 改动方向
[基于问题定义推导出的改动方向]

### 改动范围
| 区域 | 现状 | 目标状态 | 涉及文件 |
|---|---|---|---|
| ... | ... | ... | ... |

### 不改什么（显式排除）
- ...

### 验收标准
- [可执行的验收条件]

### 待决策（如有）
- [对齐过程中未解决的问题]

### 下一步
- 任务简单 → 直接动手 / `/dev-tdd`
- 任务复杂 → `/think-plan` 产出完整 spec
```

## Evidence / Stop / Risk

**Evidence**：scope 中的问题定义必须有用户确认，系统现状描述必须有 file:line 证据支撑，不接受纯推断。

**Stop**：Ambiguity Score <= 0.20 时停止追问。3-5 轮无法收敛时，给出 2-3 个候选 scope 让用户选择，附各自取舍。

**Risk**：
- 代码分析阶段发现需求可能触碰高风险区域（数据迁移、外部 API、权限变更）时，在 Model 阶段显式标出
- 发现需求与现有代码契约冲突时，放入待决策而非沉默选边

## 跳过条件

以下场景不需要本 skill：

- 需求已经具体到文件级 + 问题和验收方式都明确（直接动手）
- 典型 bug 修复，复现路径清晰（用 `/dev-debug`）
- 纯机械操作：改名、格式化、rebase
- 仓库级全局了解（用 `/think-map`）

Problem 阶段的快速通道：用户输入已包含明确的问题陈述（痛点 + 现状不可接受的原因 + 改动方向），一句话确认即可进入 Ground，不展开追问。

## Gotchas

- 不要跳过 Problem 直接进 Ground——先搞清楚问题，再去代码里找证据
- 用户说"帮我改成 X"时，先确认"为什么要改成 X"，不要直接去看怎么改成 X
- Problem 阶段不读代码——过早读代码会让 agent 被实现细节锚定，偏离问题本身
- 不要把 Ground 阶段做成 think-map 级别的全仓库扫描；只读问题相关的代码
- 不要跳过 Ground 直接问用户——这和 think-refine 没区别，失去了代码锚定的价值
- 不要把 scope 写成 spec——scope 回答"解决什么问题/改什么/不改什么"，spec 回答"怎么改"
- 3-5 轮追问无法收敛时给候选方案让用户选，不做无限循环
- 用户说"你看着办"时给推荐 + 兜底，不真的随便办

## 关联技能

- 仓库全局了解 → `/think-map`
- scope 确定后需要完整方案 → `/think-plan`
- scope 确定后任务简单可直接开发 → `/dev-tdd`
- 技术不确定需要调研 → `/think-research`
- 连续失败 2 次 → `/think-unstuck`
