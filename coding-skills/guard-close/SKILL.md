# Close

用于回答一个经常被忽略的问题：现在应该继续，还是该停了。

## 1. 触发时机

出现以下任一信号时使用：

- 准备回复“已完成”“先停在这里”“可以交付了”
- 已达到主要目标，但还在顺手补统一、补文案、扫边角
- 做着做着开始扩到用户没要求的相邻工作
- 想说“如果还需要，我可以继续做 X / Y / Z”

## 2. 先核对任务合同

先把当前工作拆成三层：

1. **用户明确要求** — 本轮交付真正要完成的内容
2. **为满足要求而必需的验证/修复** — 不做就不能声称完成
3. **相邻优化 / 顺手改进** — 模型自己想到的增强、一致性整理、锦上添花项

规则：

- 只有前两层属于当前任务
- 第三层默认进入”可选 backlog”，不能自动继续执行
- 不要把”我顺手想到的更完整方案”伪装成”用户原本就要求的内容”
- 如果上游 spec 中存在 `# spec-contract` YAML 块，用其 checks 和 non_goals 作为”用户明确要求”的基准，而不完全依赖记忆

Traceability 检查：每个已做改动应归入“用户明确要求”“为满足要求必需”“验证/cleanup 必需”。归不进去的改动属于 `Polish` / `Adjacent`，默认不继续扩大。

## 3. 停止前门禁

依次检查：

- 用户原始要求是否已完成
- 必要验证是否已完成
- 是否仍有 P0 / P1 风险
- 是否还有已经承诺但未闭环的事项
- 是否仍有无法追溯到用户请求、必要验证或 cleanup 的改动

只要以上任一答案是“否”，裁决就不能是“现在停止”。

## 4. 剩余工作分类

将剩余事项归入以下类别：

| 类别 | 含义 | 处理方式 |
|------|------|---------|
| Blocking | 不做就不能交付 | 必须继续 |
| Risk | 主目标已达成，但有明确风险需披露 | 可“带风险停止” |
| Polish | 风格统一、文案润色、边角一致性 | 默认停止，列入可选 backlog |
| Adjacent | 相邻但未被请求的新方向 | 默认停止，列入可选 backlog |

## 5. ROI / Stop Rule

出现以下信号时，优先考虑停止而不是继续深挖：

- 后续主要是风格统一、补说明、继续扫更多边角
- 对主目标没有明显增益
- 新工作来自 agent 自己扩 scope，而非用户要求
- 继续投入主要是在降低“不完美焦虑”，不是在关闭真实风险

规则：

- **投入产出低** 只能用于处理 `Polish` / `Adjacent`
- 不能用“ROI 低”逃避验证、掩盖风险、或跳过已承诺事项

## 6. 裁决规则

结尾必须给出明确裁决，只允许三种：

- **现在停止** — 目标完成、验证通过、无阻断风险、后续多为 `Polish` / `Adjacent`
- **带风险停止** — 目标完成，但仍有非阻断风险；必须写清影响
- **必须继续** — 仍有未完成要求、未验证项、P0 / P1 风险、或已承诺未闭环事项

## 7. 输出格式

```md
## Close Decision

### 当前目标
- ...

### 已完成
- ...

### 验证状态
- [x] ...
- [ ] ...

### Scope traceability
- pass / risk / fail：...

### 未继续展开的内容
以下属于可选 backlog，不影响当前交付：
- ...

### Verdict
- 现在停止 / 带风险停止 / 必须继续
- 理由：...
```

## 8. Out-of-scope 记录（可选）

当裁决为”现在停止”或”带风险停止”，且 `Adjacent` 类别中存在**重复出现过、已被裁决拒绝、未来可能再次被提出**的 enhancement 时，建议用户记录到项目的 `docs/out-of-scope/` 或 `.out-of-scope/` 目录。

规则：
- 不默认创建文件；只在满足上述条件时建议，由用户决定是否记录
- 按 concept 组织（不按 issue 编号），一个被拒绝的 concept 一个文件
- 记录内容：需求是什么、为什么不做、相关决策引用
- Bug 不适用；只用于 enhancement / feature request

## 9. Gotchas

- 不要把”可选 backlog”包装成”当前还没做完”
- 不要用”ROI 低”逃避应做验证
- 不要把真实风险降格成”以后再说”
- 不要把用户没要求的相邻工作自动并入当前范围
- `guard-close` 是裁决 skill，不是 `/guard-verify`、`/guard-review`、`/guard-ship` 的替代品
