# Refs 更新后的仓库优化方案

## TL;DR

[推断] 本轮 refs 更新后，最值得做的不是继续增加自然语言规则，而是把 refs 里反复出现的三类趋势落成可验证机制：skill routing 可评测、plugin/asset package 可校验、UI/MCP/context 风险可确定性扫描。推荐按 5 个垂直切片推进，先做 routing cases 与 verifier，再扩展 plugin/asset 校验，随后补 MCP/browser safety、UI lint、context state 边界和 refs metadata 修复。

### 关键分歧

- [推断] “引入完整 runtime” vs “吸收校验模式”：`open-design` 展示了完整 daemon/marketplace/plugin runtime，但本仓库当前核心是 agent skills / commands / hooks。推荐只吸收 manifest validation、author-side lint、event observability，不复制 daemon 或 marketplace。
- [推断] “合并技能减少数量” vs “保留细分技能并加 routing 测试”：`tw93/Waza` 有轻量 bundle 倾向，但本仓库已有 49 个细分技能。推荐不要为模仿 8-skill hard cap 而合并，先用 routing cases 降低混淆。
- [推断] “LLM critique” vs “deterministic lint”：UI 质量不能只靠 grep，但 grep/DOM/static scanner 能先拦截可复现问题。推荐 deterministic lint 负责候选证据，`fe-ui-critique` 负责设计判断。

## Research: 推荐方向与取舍

### 证据地图

| 关键结论 | 来源类型 | 证据强度 | 不确定点 |
|---|---|---|---|
| Routing 应该变成可评测接口 | 本仓库 refs-details | 高 | 具体 case schema 需设计 |
| Plugin/asset 应先做静态 manifest 校验 | 本仓库 refs-details | 中高 | 当前仓库 plugin asset 结构尚未统一 |
| MCP/browser safety 应进入 review checklist | 本仓库 refs-details + `agent-browser` 当前内容 | 中高 | `agent-browser` CLI 的实际 schema 行为未在本计划中运行验证 |
| UI lint 应从文档清单升级为可运行 scanner | 当前 `fe-ui-lint-artifact` skill + refs 趋势 | 中 | 规则 false positive 需要人工校准 |
| Context state 需要项目隔离、脱敏、TTL/cleanup | `mksglu/context-mode` refs-details + 本轮 hook 修复背景 | 中高 | 当前 `.agent-state` 生命周期策略需进一步读代码后确认 |
| refs metadata 异常应单独修 | 本轮刷新记录 | 高 | 具体异常根因需读 `.gitmodules` 和对应 path 后确认 |

### 推荐方案

推荐采用“验证机制优先”的分阶段方案：

1. 先新增 routing cases / holdout，并让 `verify_skills.py` 能检查关键 skill 触发说明与 case registry。
2. 再扩展 plugin/agent asset validator，校验 `.kilo/agent`、commands、plugin-like manifest、refs metadata 的基本契约。
3. 再把 MCP/browser safety 和 UI deterministic lint 变成可执行检查或明确 checklist。
4. 最后处理 context state 生命周期和 refs metadata repair。

推荐理由：

- [推断] 这条路径把 refs 趋势落到当前仓库已有 `scripts/verify_skills.py`、`scripts/tests/*`、`skills/*/SKILL.md`，不引入平行 runtime。
- [推断] 它先解决 routing 漏触发这一类高频 agent friction，再解决 packaging / UI / context / refs hygiene。
- [推断] 每一阶段都有单独 verifier，可以避免“写了很多文档但行为没有变”。

### 方案矩阵

| 方案 | 适用条件 | 优点 | 代价 | 脆弱假设 | 裁决 |
|---|---|---|---|---|---|
| A. 只更新文档/checklist | 希望最小改动 | 快、风险低 | 不改变 agent 行为，也难发现回归 | If agent 会主动遵守长文档 does not hold, 漏触发继续发生 | 不推荐 |
| B. verifier-first 静态校验 | 当前仓库已有 `verify_skills.py`，且 refs 趋势可静态表达 | 可测试、可 review、影响范围可控 | 需要设计 case schema 和测试 | If routing cases 不能代表真实 session does not hold, verifier 可能过拟合 | 推荐 |
| C. 引入完整 plugin/runtime/daemon | 需要产品化 marketplace 或 UI 工作台 | 功能完整，长期扩展强 | 范围过大，易产生平行体系 | If 本仓库定位仍是 skills/hooks/docs does not hold, runtime 会成为维护负担 | 暂缓 |
| D. session-driven 动态评测 | 有足够 session 样本和隐私边界 | 更贴近真实行为 | 数据脱敏、抽样、判定成本高 | If session 样本偏差大 does not hold, 结论不稳定 | 第二阶段补充 |

### 不要自造轮子

- 不要自造完整 marketplace / daemon：先借鉴 `open-design` 的 manifest validation 和 event observability 概念。
- 不要自造浏览器自动化 runtime：继续复用 `agent-browser` / MCP/browser tools，把安全 checklist 和验证边界做好。
- 不要自造大型 UI 视觉引擎：先做 grep/DOM/static lint，必要时后续再接 `agent-browser` 截图和 `fe-ui-visual-iterate`。

### 运行边界

- 权限与凭据：本计划本身不需要凭据；后续联网 refs refresh、browser/MCP 测试、外部 URL 检查必须走明确授权。
- 写操作与副作用：后续实现会改 scripts、tests、skills、docs；不应修改 refs submodule pointer，除非用户明确批准。
- 成本与配额：默认无云成本；若后续引入 browser/MCP live test 或 Web fetch，应单独标注网络和 rate limit。
- 回滚：每个阶段以独立文件集提交，失败时按阶段回滚。
- Acceptance verifier：运行 routing cases、plugin/asset validation、UI lint fixture、context state tests、refs metadata check，并用一份 `/skill-maintenance` dry run 证明信号进入报告。

## 目标与非目标

### Goals

- 把 refs 更新后的高价值趋势转成仓库内可验证机制。
- 降低 skill 漏触发、asset 结构漂移、UI slop、MCP/browser 边界遗漏、refs metadata 异常的复发概率。
- 让 `/skill-maintenance` 后续报告能引用 deterministic evidence，而不是只引用人工观察。

### Non-goals

- 不引入完整 plugin marketplace / daemon / GenUI runtime。
- 不大改 `agents/AGENTS.md`。
- 不移动或更新 submodule pointer。
- 不为 legacy `se-*` 做 alias。
- 不把所有 refs 趋势一次性吸收到实现中。

### Constraints

- 行为变更和脚本增强必须走 TDD。
- 每个阶段必须有 inner-loop verifier 和 acceptance verifier。
- 涉及联网、外部目标、浏览器登录态、secrets 的实现必须先经过对应 guard。
- 保持 surgical changes，避免顺手重构不相关文件。

## 详细实施计划

### Phase 1: Routing Cases / Holdout

目标：把 skill routing 从“文案感觉”变成可测试接口。

建议文件：

- 新增 `docs/skill-routing-cases.md` 或 `scripts/fixtures/skill_routing_cases.json`。
- 修改 `scripts/verify_skills.py`。
- 新增或修改 `scripts/tests/test_skills_registry.py`。

Contract cases：

| Input pattern | Expected skill | Reject / ambiguity |
|---|---|---|
| 修 bug、异常、复现不稳定 | `dev-debug` 或 `dev-tdd` handoff | 不能只走 `think-plan` |
| 新功能、行为变更 | `dev-tdd` | 纯文档/配置不强制 |
| URL/PDF/GitHub 页面读取 | `web-read` | 不直接临时 fetch 多源后跳过 survey |
| PR/code review | `guard-review` | agent config/harness 审查需二级路由 |
| UI artifact / HTML / CSS slop | `fe-ui-lint-artifact` 或 `fe-ui-critique` | 不用泛泛“AI 感” |
| 远程/部署/refs fetch | `guard-gitops` | 只读查询可降级 |

Verification：

- Inner-loop：`python3 -m unittest scripts.tests.test_skills_registry -v`。
- Acceptance：`python3 scripts/verify_skills.py` 输出 routing cases summary，并能对至少 10 个 fixture case 给出 pass/fail。

### Phase 2: Plugin / Agent Asset Validator

目标：吸收 `open-design` / `ChromeDevTools` / `awesome-copilot` 的 manifest validation 趋势，但不复制 runtime。

建议文件：

- 修改 `scripts/verify_skills.py` 或新增 `scripts/verify_agent_assets.py`。
- 修改 `scripts/tests/test_install_hooks.py` 或新增专门测试。
- 覆盖 `.kilo/agent/*.md`、`commands/*.md`、可能的 `.codex-plugin/plugin.json`、`.claude-plugin/plugin.json`、`.agents/plugins/marketplace.json`。

Contract cases：

- `.kilo/agent/*.md` 必须有 `description`、`mode`、`model`、`permission` 或明确豁免。
- command frontmatter 必须有 `description`；有参数时有 `argument-hint` 或说明无需参数。
- plugin manifest 如存在，必须校验 JSON、name/version/source/entry path、相对路径不逃逸仓库。
- 对 version pinning 给 warning，不在第一阶段强制 blocker。

Verification：

- Inner-loop：新增 fixture 测试覆盖 missing field、bad path、valid asset。
- Acceptance：`python3 scripts/verify_skills.py` 或新 verifier 可报告 assets validated count。

### Phase 3: MCP / Browser Safety Checklist

目标：把 ChromeDevTools ref 的工具契约和路径边界落入 browser/MCP 使用规范。

建议文件：

- `skills/agent-browser/SKILL.md`
- `skills/guard-secure/SKILL.md`
- 可选新增 `docs/software-engineering-research/mcp-browser-safety.md`，仅当内容超过 skill 内可维护范围时创建。

Contract cases：

- filePath / downloads / screenshots 需说明 realpath 或 workspace boundary。
- unknown args / unsupported CLI options 应 fail-loud，不默默继续。
- browser session / daemon 需有 shutdown / cleanup 指令。
- 登录态、cookies、auth state 文件继续遵守 secret handling。

Verification：

- Inner-loop：`python3 scripts/verify_skills.py`。
- Acceptance：人工检查 `agent-browser` 的 auth、download、evaluate/script、cleanup 段落是否覆盖 contract；如未来有 CLI 可测，再补 command smoke test。

### Phase 4: UI Deterministic Lint 增强

目标：让 UI slop findings 可复现。

建议文件：

- `skills/fe-ui-lint-artifact/SKILL.md`
- 可选新增 `scripts/scan_ui_artifact.py`
- 新增 `scripts/tests/test_scan_ui_artifact.py`

Contract cases：

- 默认紫/靛/青渐变、emoji filler、placeholder copy、虚构指标、外部占位图、overflow/truncate 高风险使用。
- 每个 finding 输出 `priority`、`id`、`file:line`、`snippet`、`fix`。
- 允许 false-positive suppression：theme token、design-system-approved、decorative clipping。

Verification：

- Inner-loop：fixture HTML/CSS 触发 P0/P1/P2；合法 theme token 不报。
- Acceptance：对一个人工构造 artifact 输出稳定 finding 表，`fe-ui-lint-artifact` 文档与 scanner 输出契约一致。

### Phase 5: Context State / Refs Metadata Hygiene

目标：补齐本轮已发现的状态和 refs hygiene 风险。

建议文件：

- `scripts/hooks/context_state.py`
- `scripts/tests/test_hook_context_state.py`
- `scripts/skill_maintenance_collect.py`
- `scripts/tests/test_skill_maintenance_collect.py`

Contract cases：

- context state 具备 project scope、redaction、TTL/cleanup 或明确 no-cleanup rationale。
- collector 发现 `.gitmodules` path 与实际 submodule mapping 不一致时，输出 deterministic finding。
- refs fetch 失败不中断全局 collector，但必须记录 path、error、建议修复。

Verification：

- Inner-loop：`python3 -m unittest scripts.tests.test_hook_context_state scripts.tests.test_skill_maintenance_collect -v`。
- Acceptance：构造缺 mapping ref fixture，collector 输出 `refs-metadata-warning`。

## 风险与防线

| Risk | Impact | Mitigation | Verification |
|---|---|---|---|
| Routing cases 过拟合少量示例 | 真实任务仍漏触发 | 从 session 分析、用户命令、refs trend 三类来源取 case | Holdout cases 不与实现逻辑共用文本 |
| Verifier 变得太吵 | agent 忽略 warning | severity 分层：blocker / should / observe | collector 输出按 severity 聚合 |
| Plugin validator 范围失控 | 引入 marketplace/runtime 债务 | 第一阶段只做静态字段和路径校验 | 不新增 daemon，不新增 package manager |
| UI scanner false positives | 修复成本高 | suppression 与 context read 必需 | 测合法 token 不报 |
| Context cleanup 误删有用恢复状态 | compact 恢复变弱 | TTL 策略先 observe，再 apply | 测 legacy recovery 与 cleanup 不冲突 |
| Refs metadata repair 误动 submodule pointer | 破坏 refs SSOT | 第一轮只检测不修复；修复需单独批准 | `git diff --submodule` 人工核对 |
